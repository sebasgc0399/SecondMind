# Gotchas — i18n (i18next, catálogos, plurales)

## `count` como variable de interpolación activa pluralización — clasificar el sitio ANTES de elegir el nombre

En i18next, `t('key', { count: n })` no es una interpolación más: `count` fuerza la resolución plural (`key_one`/`key_many`/`key_other` vía Intl.PluralRules) y el `extract` del CLI genera esas variantes en los catálogos. Para sitios donde el texto alrededor del número NO cambia ("Hub: N conexiones", "Ver todas las N" — plural fijo del copy actual, invariante del baseline F58), usar `count` habría cambiado el contrato de la key y generado variantes vacías parásitas. Detectado en F58 F2.2 (HubsCard/ReviewCard.seeAll).

**Regla operativa:** antes de interpolar un número, clasificar el sitio:

- El texto alrededor **cambia** con el número ("1 tarea pendiente" / "2 tareas pendientes") → `{ count }` + formas `_one`/`_many`/`_other` en el catálogo (las tres: el español moderno tiene `many` en CLDR y dejarla vacía arriesga key cruda — poblarla con el valor de `_other`).
- El texto alrededor **no cambia** → cualquier otro nombre de variable (`{{total}}`, `{{n}}`), nunca `count`.

Tras cada `extract`, revisar el diff del catálogo: variantes plurales nuevas que no se diseñaron = un `count` mal clasificado en el código.

## Una key no puede ser string Y objeto a la vez — colisión al anidar bajo una key plana existente

i18next resuelve por path: si `inbox.process` ya es un string ("Procesar"), no podés tener además `inbox.process.title` (objeto) — el nodo `process` no puede ser hoja y rama simultáneamente. El catálogo se corrompe silenciosamente (una de las dos resuelve a `[object Object]` o a la key cruda). Detectado en F58 F2.4: el botón "Procesar" (`inbox.process`) chocaba con el sub-árbol del processor (header "Procesando Inbox", "Salir", etc.).

**Regla operativa:** antes de poblar un sub-árbol jerárquico, chequear si el segmento padre ya existe como key plana. Si choca, renombrá uno de los dos — en F2.4 el sub-árbol pasó a `inbox.processView.*` y el botón plano quedó `inbox.process`. El gotcha escala con la profundidad de la jerarquía: a más dominios con keys cortas reutilizadas (`*.title`, `*.edit`, `*.process`), más superficie de colisión. **Vigilancia para F2.5 (editor):** las jerarquías de slash commands / menús / tooltips reusan segmentos como `heading`, `list`, `link` — verificar colisión plana-vs-objeto al diseñar las keys, no después del extract.

## `<Trans>` solo para elementos DENTRO de la frase — iconos/elementos hermanos usan `t()` plano

`<Trans>` (react-i18next) se necesita cuando un elemento React vive INLINE dentro del texto traducible: `<kbd>`, `<strong>`, `<a>`, `<1>...</1>` que parten la frase y cuyo orden puede cambiar por idioma (ej. `inbox.empty.hint` = "Captura una idea con `<kbd>Alt+N</kbd>` desde cualquier pantalla"). NO se necesita cuando el elemento es HERMANO del texto, aunque estén en el mismo contenedor flex: `<div className="flex"><Sparkles/> Sugerencia AI</div>` son dos hijos hermanos del div, no un elemento dentro de la frase → `t('...', 'Sugerencia AI')` plano basta. Detectado en F2.5: de 14 sitios pre-marcados como `<Trans>` por el inventario, ~0 lo eran realmente (todos eran icono lucide + texto hermanos). Heurística: ¿el elemento PARTE la frase (texto antes Y después del elemento)? → `<Trans>`. ¿El elemento está ANTES o DESPUÉS de todo el texto? → hermano, `t()` plano. **Vigilancia onboarding (F2.6):** texto con formato (negritas en pasos, links inline) — clasificar cada sitio antes de migrar; los pasos del checklist con `<strong>` embebido SÍ son `<Trans>`, un ícono de paso + label NO.

## Const i18n consumida por React Y por módulo no-React: `buildX(t)` con doble wrapper

Cuando una const con strings (array de items, Record de labels) la consumen DOS caminos —componentes React Y código module-scope (extensiones TipTap, listeners, factories que corren fuera de React)— el patrón const→hook (navItems) rompe el camino no-React (no puede llamar un hook). Solución: extraer `buildX(t: TFunction)` como fuente ÚNICA, exponer dos wrappers: `useX()` (hook, `t` de `useTranslation` — para React, reactivo al idioma) y `getX()` (`i18n.t.bind(i18n)` — module-scope, lee el idioma vigente por llamada). Combina navItems-pattern (hook) + el patrón `i18n.t` directo en módulos no-React (formatDate.ts, wikilink.ts). Sentado en F2.5: `slashMenuItems` const → `buildSlashMenuItems(t)` + `useSlashMenuItems()` (SlashMenu) + `getSlashMenuItems()` (extensión `slash-command-suggestion.ts`). Si una función pura recibía la const global, refactorizarla para recibir los items como argumento (`filterSlashMenuItems(items, query)`), y cada consumidor inyecta los suyos. **Vigilancia F2.6:** si onboarding comparte lógica de pasos entre un componente y un helper module-scope, aplica el mismo patrón.

## `<Trans values>` con un campo `string | null` rompe el build (TS2322) — coalescer a `''`

`<Trans values={{ email: user.email }}>` falla la compilación con `TS2322: Type 'string | null' is not assignable to type 'string'` cuando el campo viene de un shape opcional (típico de Firebase Auth: `user.email` es `string | null`). El JSX plano `{user.email}` toleraba el `null` (React no renderiza `null`), pero el objeto `values` de `<Trans>` exige `Record<string, string>` estricto → no hay coerción implícita. Fix mínimo y equivalente al comportamiento previo: `values={{ email: user.email ?? '' }}` (un email ausente deja el `<1></1>` vacío, igual que el `{null}` anterior no pintaba nada). Detectado en F2.6 (verify-email/page.tsx con `user.email`). **Sutil:** el error NO lo agarra el lint ni el `extract`/`types` del CLI — solo aparece en `tsc -b` (paso de `npm run build`), así que un dominio que migra `<Trans>` con campos de Auth/Firestore nullable puede pasar tests+lint verdes y romper recién en build. Regla: al interpolar un campo de un shape externo dentro de `<Trans values>`, asumir nullable y coalescer (`?? ''`) salvo que un guard previo ya estreche el tipo a `string`.

## El check tsc de keys solo aplica al form sin defaultValue

Con resources tipados, i18next valida la key contra la unión estricta SOLO en los overloads `t(key)` y `t(key, options)`. El overload `t(key, defaultValue, options?)` (defaultValue posicional como 2º arg string) tipa la key como `string` **loose** → un typo de key **NO lo caza tsc**. Verificado en F58 F3.2 rompiendo el verificador en ambos sentidos: `t('errors.betaFullX', 'default')` pasó `tsc -b --force` con exit 0; `t('errors.rateLimitedNOPE')` (sin defaultValue) falló con **TS2345** ("Did you mean errors.rateLimited?"). **Corrige el hallazgo impreciso de F1** ("typo en key = error de tsc"): eso aplica SOLO al form sin defaultValue.

El repo usa `t(key, defaultValue)` por convención (mapProcessError, entityLabels, ObjectiveCard, mapCfError, authErrors) — **deliberado**: el defaultValue siembra el copy es vía el extractor AST y da el fallback en runtime (copy correcto aunque la key falte). El canje es consciente: se renuncia al check tsc de keys a cambio de esas dos protecciones. **NO migrar a `t(key)` solo para recuperar el check** — perdería el defaultValue (mal canje: una protección por otra).

**Protecciones en el form con defaultValue (reemplazan al check tsc):**

1. El extractor ve la key literal → no la purga en F4 (`removeUnusedKeys`).
2. El defaultValue se renderiza en runtime si la key falta (nunca key cruda).
3. **Verificación de catálogo** (el control que sí caza un typo de key): grep de las keys usadas en código vs las del catálogo, salida vacía. Ej. F3.2: `grep -ohE "errors\.[a-zA-Z]+" src/lib/cfError.ts src/lib/authErrors.ts | sort -u` vs `Object.keys(es.errors)` → `comm -23` vacío (30/30 keys existen).

**Done criterion del arco i18n — sustitución (para F4 y futuros):** en dominios que usan `t(key, defaultValue)`, el criterio "probar el verificador rompiéndolo" (typo de key → tsc falla) **NO aplica** — reportarlo como verde es falso-verde. Sustituir por la **verificación de catálogo** (`comm -23` keys-usadas-en-código vs keys-del-catálogo, salida vacía). El gate tsc sigue siendo válido para typos en dominios/sitios que usan el form `t(key)` sin defaultValue.
