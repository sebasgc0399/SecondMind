# Gotchas — i18n (i18next, catálogos, plurales)

## `count` como variable de interpolación activa pluralización — clasificar el sitio ANTES de elegir el nombre

En i18next, `t('key', { count: n })` no es una interpolación más: `count` fuerza la resolución plural (`key_one`/`key_many`/`key_other` vía Intl.PluralRules) y el `extract` del CLI genera esas variantes en los catálogos. Para sitios donde el texto alrededor del número NO cambia ("Hub: N conexiones", "Ver todas las N" — plural fijo del copy actual, invariante del baseline F58), usar `count` habría cambiado el contrato de la key y generado variantes vacías parásitas. Detectado en F58 F2.2 (HubsCard/ReviewCard.seeAll).

**Regla operativa:** antes de interpolar un número, clasificar el sitio:

- El texto alrededor **cambia** con el número ("1 tarea pendiente" / "2 tareas pendientes") → `{ count }` + formas `_one`/`_many`/`_other` en el catálogo (las tres: el español moderno tiene `many` en CLDR y dejarla vacía arriesga key cruda — poblarla con el valor de `_other`).
- El texto alrededor **no cambia** → cualquier otro nombre de variable (`{{total}}`, `{{n}}`), nunca `count`.

Tras cada `extract`, revisar el diff del catálogo: variantes plurales nuevas que no se diseñaron = un `count` mal clasificado en el código.
