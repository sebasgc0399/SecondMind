## Objetivo

Cerrar el ciclo del soft delete que F18 dejó abierto: las notas eliminadas ahora viajan a una papelera visible (segmento "Papelera" dentro de `/notes`) donde pueden restaurarse o eliminarse para siempre, con limpieza automática a X días configurables (Nunca / 7 / 15 / 30, default 30) y cleanup completo de embeddings + links bidireccionales vía CF onDelete. Además, una nota eliminada deja de aparecer en el grafo y deja de ser accesible vía URL directa `/notes/{id}`.

Habilita la primera infra de preferencias del usuario (`users/{uid}/settings/preferences`) y la primera Cloud Function scheduled del proyecto (`autoPurgeTrash`).

## Lecciones

- **`baseRepo.remove(id)` ya respetaba sync→async (`delRow` antes de `await deleteDoc`).** F2 quedó en una línea de wrapper. El SPEC original proponía implementarlo manualmente — siempre auditar el factory de baseRepo antes de duplicar lógica. Trade-off conocido del orden sync-antes-async para hard delete: si `deleteDoc` falla por red, la row desaparece localmente pero el `onSnapshot` del persister la re-hidrata al reconectar y la nota "desaparece y reaparece". Aceptable por consistencia con resto de repos; el user puede reintentar trivialmente.
- **Firestore RECHAZA single-field indexes en `firestore.indexes.json`** con `"this index is not necessary, configure using single field index controls"`. Los single-field indexes están auto-habilitados en COLLECTION + COLLECTION_GROUP scope por default. Solo se declaran composite indexes en JSON. La query `collectionGroup('notes').where('deletedAt', '>', 0)` funciona con el implícito. La entry `"firestore.indexes": "firestore.indexes.json"` en `firebase.json` sigue siendo necesaria para que `firebase deploy --only firestore:indexes` encuentre el archivo (sino falla silencioso).
- **`firestore.rules` catch-all `match /users/{userId}/{document=**}`cubre cualquier sub-colección nueva** sin cambios. F1 NO modificó rules y aún así protegió`users/{uid}/settings/preferences` correctamente. Verificación cross-user via collectionGroup queries la enforce Firestore por doc-path: un cliente autenticado como uidA solo puede leer sus propias notas. Cualquier PR futuro que toque rules debe re-correr el test cross-user antes de mergear.
- **React Router NO maneja anchor scroll automáticamente.** Implementación canónica: `useEffect(() => { if (location.hash !== '#X') return; requestAnimationFrame(() => document.getElementById('X')?.scrollIntoView({ behavior: 'smooth', block: 'start' })); }, [location.key, location.hash])`. Dep en `location.key` (no `.hash`) para re-trigger en navegaciones repetidas al mismo hash. `requestAnimationFrame` porque el DOM puede no estar pintado al primer tick post-navigate. `scroll-mt-14` en la sección compensa MobileHeader sticky de `h-14`.
- **Backwards-compat de cron scheduled con datos pre-feature: hardcodear `FIRST_DEPLOY_TS`.** F18 dejó notas con `deletedAt > 0` desde días atrás; el primer run del cron F4 las habría purgado masivamente. Mitigación: `const F19_FIRST_DEPLOY_TS = Date.UTC(2026, 3, 26)` + lógica `effectiveDeletedAt = max(deletedAt, F19_FIRST_DEPLOY_TS)`. Notas pre-feature obtienen grace period equivalente al config desde el deploy. NO modificar el timestamp después del deploy. Patrón aplicable a cualquier futuro scheduled CF que opere sobre datos retroactivos.
- **Primer `onSnapshot` a doc único del proyecto requiere convención propia.** Los demás `onSnapshot` viven dentro del persister de TinyBase (collections completas). Para preferences, centralizar en `lib/X.ts` con cache module-level + dedupe por uid (patrón `embeddings.ts`); el hook React es solo wrapper sobre `subscribeX(uid, cb): unsubscribe`. Esto deja la puerta abierta a migrar a tabla TinyBase via persister cuando crezca el set de prefs (>3 campos), sin tocar callsites.
- **Primer scheduled CF del proyecto: `firebase-functions/v2/scheduler`.** Schedule cron tradicional `'0 3 * * *'` con `timeZone: 'UTC'` explícito (más portable que sintaxis App Engine `'every day 03:00'`). Memory `256MiB`, timeout `540s` (máx scheduler v2), `retryCount: 0`. Habilita la API `cloudscheduler.googleapis.com` automáticamente al desplegar. Verificación post-deploy: `firebase functions:list` muestra trigger `scheduled`.
- **CFs de write disparadas por delete son no-op-safe** si chequean `if (!event.data?.after?.data())` al inicio. Tanto `autoTagNote` como `generateEmbedding` ya hacen este check. Cuando F4 borra una nota, `autoTagNote` y `generateEmbedding` se disparan junto con `onNoteDeleted` (todos en `onDocumentWritten`/`onDocumentDeleted`) — los primeros salen sin hacer nada porque `after` es `undefined`. Garantizar este guard en cualquier futura CF `onDocumentWritten` que opere sobre la misma colección.
