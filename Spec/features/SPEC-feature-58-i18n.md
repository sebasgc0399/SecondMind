## Objetivo

Al cerrar esta feature el usuario puede usar SecondMind completa en español o inglés: la UI, los textos generados por la IA (títulos, summaries, tags sugeridos) y los mensajes de error siguen el idioma elegido en Settings (detectado de `navigator.language` la primera vez). El copy español actual no cambia visualmente.

Stack runtime: react-i18next 17.0.8 + i18next 26.3.1 + Intl nativo; tooling dev-only i18next-cli 1.62.0. Single namespace `translation`, keys jerárquicas por dominio, catálogos estáticos al bundle (offline PWA + 3 frentes nativos), `fallbackLng: 'es'`.

## Lecciones

- **`instrument` del i18next-cli es DETECTOR, nunca aplicador.** En F2.1 su AST writer corrompió código real (nombre de función partido, `t()` inyectado dentro de un `className`) y emitió `i18next.t()` a module-scope (no reactivo al switch de idioma), además de false positives (classNames Tailwind, `⌘K`, marca) y false negatives (aria-labels, `title=` en ternarios). Protocolo vigente: correr `instrument --dry-run` solo para el inventario de sitios, **descartar el diff completo** (`git checkout --`), y editar a mano con keys jerárquicas + `defaultValue` inline. Recall confirmado ~80-90% → la revisión manual NO es opcional. Keys SIEMPRE literales dentro del componente (`labelKey` dinámico rechazado por extract y por los tipos: `t(string)` no compila).
- **Unificar helpers de formato solo donde el output coincide 1:1; si no, localizar in-place.** La invariante "es idéntico al baseline" (D8) manda sobre la consistencia. Caso: `formatDeadline` de ObjectiveCard ("faltan 5 días") difería de lo que daría `formatRelative` ("hace X / dentro de X") → se localizó con keys propias en vez de unificar. La unificación es mejora, no requisito; se evalúa caso por caso comparando el output exacto.
- **Plural `_many` en es es obligatoria con CLDR moderno** y el `extract` la genera — poblarla con el valor de `_other`, nunca vacía (key vacía + fallback vacío arriesga key cruda en pantalla). Regla dura inversa para en: **cero `_many` en el catálogo en** (CLDR en = one/other; las `_many` son es-only).
- **`locale` aditivo con default → NO bumpear `PREFERENCES_SCHEMA_VERSION`.** El patrón del repo (precedente F46/F49, sentinel anti-bump en `preferences.test.ts`) es parse defensivo sin bump; bumpear purgaría las prefs reales (`parsePrefs` devuelve defaults ante mismatch). La regla del gotcha se precisó para distinguir aditivo (no bump) vs breaking (bump). Tipo `'es' | 'en' | null` donde `null` = "nunca elegido", distingue elección de detección y dispara el write eager que F3.1 lee server-side.
- **Tipos i18n: typo en key = error de `tsc`, pero solo en la forma sin `defaultValue`.** Verificado rompiéndolo deliberadamente (gotcha "probar el verificador"). Limitación escalada a `gotchas/i18n.md`.
- **El install de i18next desincronizó react/react-dom** (19.2.7 vs 19.2.5; React exige igualdad exacta) — tercera instancia del gotcha `deps-build`. Cualquier `npm install` que mueva el lockfile reincide; verificar `resolve.dedupe` al reiniciar el dev server.
- **Strings en INGLÉS pre-existentes en la UI es son excepciones APROBADAS a la invariante de baseline:** se homogeneizan al canon español vía catálogo en su dominio (Settings→Ajustes, note-types Fleeting/Literature/Permanent→Fugaz/Literatura/Permanente, "Area"→"Área"). El diff contra baseline en ESOS puntos es deseable; cualquier otra diferencia sigue siendo NO-GO y se agrega a la lista cerrada solo con aviso explícito a Sebastián.
- **Reconciliación de versión al cerrar un arco largo:** 0.5.0 ya había shipeado antes de arrancar el arco i18n, así que los 102 commits unreleased salieron como 0.5.1 — el número de release se deriva del estado real de prod, no del plan. Gotchas i18n del arco ya escalados a `Spec/gotchas/i18n.md` (flip `removeUnusedKeys`, resolución `{{type}} note`, reorder de tiempo relativo es→en, regla `_many`).
