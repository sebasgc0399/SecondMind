## Objetivo

Dogfooding con Playwright sobre 9 páginas × 2 viewports destapó cuatro categorías de issues visuales que degradaban la percepción de calidad sin afectar funcionalidad: scrollbar fantasma en tabs de Tareas (`overflow-x-auto` sin `overflow-y-hidden` explícito dispara scrollbar vertical en Chrome/Edge Windows), loaders `<p>Cargando...</p>` que violaban la regla CLAUDE.md "skeleton siempre, spinner/texto nunca", ~22 strings con voseo rioplatense mezclado con imperativo peninsular + 8 typos de tildes recurrentes, y duplicación de título en mobile (MobileHeader sticky + H1 interno de cada página). F15 resolvió los 4 en un pase coordinado 100% client-side, deploy solo-hosting sin bump de versión.

## Qué se implementó

- **F15.1 — Scrollbar fantasma suprimido.** [src/app/tasks/page.tsx:109](../../src/app/tasks/page.tsx#L109): `<nav className="... overflow-x-auto">` → `overflow-x-auto overflow-y-hidden`. Spec CSS computa `overflow-y: auto` implícito cuando `overflow-x` es `auto`; Chrome/Edge en Windows renderean scrollbar vertical "por las dudas" aunque no haya overflow real. Dos clases juntas suprimen el fantasma sin romper scroll horizontal de tabs cuando crezcan.
- **F15.2 — Shell skeleton pre-auth responsive.** Nuevo [src/components/layout/AuthLoadingSkeleton.tsx](../../src/components/layout/AuthLoadingSkeleton.tsx): shell completo con sidebar 240px + main + cards (desktop) y header h-14 + main + bottom nav placeholder (mobile), alternadas via CSS puro (`hidden md:flex` / `md:hidden`). Cero JS, cero `useBreakpoint`, dimensiones matchean el chrome real para evitar layout shift al transicionar `isLoading: true → false`. Consumido en [src/app/layout.tsx](../../src/app/layout.tsx) (reemplaza el `<p>Cargando...</p>` pre-auth). En [src/app/login/page.tsx](../../src/app/login/page.tsx) se usa un skeleton inline minimal (3 bloques centered) — NO reusa `AuthLoadingSkeleton` porque `/login` no tiene sidebar/main/nav, solo form centrado.
- **F15.3 — Imperativo neutro + tildes.** 22 strings modificadas en 15 archivos, string-only, sin tocar lógica. Voseo rioplatense (`Creá`, `Revisá`, `Escribí`, `Intentá`) → imperativo neutro (`Crea`, `Revisa`, `Escribe`, `Intenta`). Tildes faltantes en palabras recurrentes: `revisión`, `próxima`, `aún`, `conexión`, `más`, `ningún`, `periódica`, `sincronizarán`, `mañana`, `días` (los dos últimos encontrados en [src/components/editor/ReviewBanner.tsx:15-16](../../src/components/editor/ReviewBanner.tsx#L15-L16) durante el scan, no en el SPEC original). Archivos afectados: `QuickCapture.tsx`, `capture/page.tsx`, `tasks/page.tsx`, `projects/[projectId]/page.tsx`, `projects/page.tsx`, `objectives/page.tsx`, `DistillIndicator.tsx`, `useAutoUpdate.ts`, `ReviewBanner.tsx` (5 strings), `SimilarNotesPanel.tsx` (2 strings), `inbox/page.tsx`, `InboxCard.tsx`, `OfflineBadge.tsx` (2 tildes en una línea), `notes/graph/page.tsx` (3 strings: `conexión`, `Ningún`, `más`), `GraphNodePanel.tsx`. Greps finales (`Creá|Revisá|Escribí|Intentá` y `revision|Proxima|aun\.|conexion|periodica|sincronizaran|Ningun nodo|mas notas|manana|dias`) = 0 matches.
- **F15.4 — H1 duplicado oculto en mobile con estrategia granular.** `MobileHeader` ya renderea `<h1>` con `getPageTitle(pathname)` — 8 páginas duplicaban el título en un H1 interno. Estrategia variable según si el header contiene siblings esenciales:
  - **4 páginas con `hidden md:block/flex` al wrapper `<header>`** (sin siblings esenciales o cubiertos por chrome mobile): `tasks` (h1 único), `settings` (subtítulo decorativo), `projects` (botón "Nuevo proyecto" — FAB cubre captura), `objectives` (botón "Nuevo objetivo" — ídem).
  - **3 páginas con restructure (`hidden md:block/flex` solo al `<h1>` o div wrapper)** preservando siblings esenciales: `habits` (flechas mes/año), `inbox` (Link "Procesar"), `notes` (botones "grafo" + "Nueva nota").
  - **`/notes/graph` agregada tardíamente (commit `6140cdb`):** el audit asumió que `MobileHeader.getPageTitle` haría prefix match sobre `/notes` y devolvería "Notas", haciendo útil el h1 interno "Grafo". E2E reveló que `/notes/graph` está en `navItems` (`Sidebar.tsx:38`) con exact match → devuelve "Grafo", causando duplicación real. Fix: `hidden md:inline` al h1 interno de graph, preservando back link + stats + fullscreen.
  - **Dashboard excepción intencional:** conserva `<h1>Buenas noches, sebastian</h1>` — copy personalizado distinto del "Dashboard" del MobileHeader, aporta valor.

## Lecciones

- **`overflow-x-auto` sin `overflow-y-hidden` explícito es el gotcha más común del scroll chrome en Windows.** La spec CSS computa `overflow-y: auto` implícito cuando `overflow-x` es `auto`/`scroll`. Chrome/Edge en Windows renderean un scrollbar vertical fantasma "por las dudas" aunque el elemento no tenga overflow real. Fix canónico: siempre parejar `overflow-x-auto overflow-y-hidden` (o `overflow-y-visible` si aplica). Aplicable a cualquier componente del proyecto con tabs scrolleables.
- **El audit con Explore agents puede tener blindspots que solo E2E destapa.** El agent que mapeó headers asumió que `MobileHeader.getPageTitle` haría prefix match para `/notes/graph` (`pathname.startsWith('/notes')` → "Notas"), pero no verificó la lista `navItems`. `/notes/graph` ahí con exact match → devuelve "Grafo" → causa duplicación. Lección: post-audit, las asunciones críticas de labeling/routing deben validarse con Playwright ejecutando el código real antes de commitear. Budget de tiempo para fixes E2E-descubiertos debe incluirse en el plan.
- **Skeletons responsivos con CSS puro escalan mejor que con JS hooks.** `<AuthLoadingSkeleton>` renderea ambas variantes desktop/mobile simultáneamente y CSS oculta la que no aplica (`hidden md:flex` / `md:hidden`). Cero runtime cost de decisión, cero flash si matchMedia tarda en resolver, cero dependencia de providers (crítico para pre-auth gate donde StoreHydrationProvider no existe todavía). Patrón reaplicable a cualquier componente responsive sin state.
- **El costo de un Plan agent extra vale la pena para re-visitar decisiones del SPEC.** El SPEC original decía "inline 15 líneas" para el shell skeleton y "ocultar `<header>` completo" para F15.4. Los Explore + Plan agents destaparon que ambas decisiones no escalaban al caso real (~70 LoC responsive, 5 headers con controles esenciales). Corregir en plan mode vs mid-codeo ahorra churn y commits exploratorios. El SPEC prescriptivo es una hipótesis inicial; el plan refinado es la ejecución validada.
- **String rename mass-scale con `Edit` tool literal es más seguro que regex replace.** 22 strings en 15 archivos, cada una con contexto distinto (interpolaciones, comillas tipográficas, ternarios singular/plural). `replace_all` habría pisado `conexiones` queriendo cambiar `conexion`. Edits literales paralelos en batch preservan la seguridad sin perder velocidad. Los greps finales (0 matches) certifican el rename completo sin falsos positivos.
