# SPEC F28 — `saveContent` retry/rollback (Registro de implementación)

> Estado: Completada abril 2026
> Commits: `89dcb34` F28.1 saveQueue primitiva, `04788c4` F28.2 integración useNoteSave, `e7c243c` F28.3 banner UI, `365d860` F28.4 flush hooks, `6b19e36` hot-fix savedBadgeVisible.
> Gotchas operativos vigentes → `Spec/ESTADO-ACTUAL.md`

## Objetivo

Cerrar el silent data loss en `notesRepo.saveContent`. Pre-F28: si `await updateDoc` fallaba por red u otro error transient, el catch en `useNoteSave.ts:90-94` reseteaba a `setStatus('idle')` (mismo estado visual que "no hay cambios pendientes"). El usuario seguía tipeando creyendo que se guardaba. Además, el `setPartialRow` ya había escrito título/preview/metadata a TinyBase sync; el persister TinyBase eventualmente propagaba esos campos a Firestore por su lado, dejando estado inconsistente: la lista de notas mostraba el preview nuevo, pero el `content` (JSON TipTap) en Firestore quedaba stale.

Solución: retry queue in-memory con backoff `[1s, 2s, 4s]` sobre 4 attempts máximos, `SaveStatus` extendido a 5 estados, banner top-of-page tras 4 fallos consecutivos con acciones [Reintentar / Descartar / Copiar contenido]. Scope acotado a `saveContent`; el factory `createFirestoreRepo` queda con la misma falla (impact menor: metadata, no contenido del usuario) como deuda futura.

## Qué se implementó

- **F28.1 — Retry queue primitiva (`src/lib/saveQueue.ts`):** factory `createSaveQueue<T>` con API `enqueue/cancel/retryNow/flushAll/subscribe/getEntry/getSnapshot/dispose`. Backoff `[1000, 2000, 4000]ms` antes de cada retry; tras 4 fallos totales (1 initial + 3 retries) status pasa a `'error'`. Detección de errores permanentes via `instanceof FirebaseError` + code en `{permission-denied, failed-precondition, invalid-argument, unauthenticated}` → fast-fail directo a `'error'` sin consumir attempts. Upsert semantics: `enqueue` con id existente reemplaza payload + executor + bumpea version, NO resetea attempts (excepto cuando estaba en `'error'` — ahí sí resetea, equivalente a retryNow desde error). Race cancel-vs-in-flight resuelto con flag `cancelled` en la entry — el executor consulta al resolver y aborta el `setStatus('synced')`. LRU cap de 50 entries con eviction por `createdAt` ascendente (excluye entries en `'syncing'`). API canónica para `useSyncExternalStore`: `subscribe(cb)` sin args + `getSnapshot()` separado. Singleton `saveContentQueue` exportado al final del módulo, tipado con `SaveContentPayload` del notesRepo (sin ciclo: notesRepo no importa saveQueue). 14 tests vitest.

- **F28.2 — Integración `useNoteSave` + `SaveStatus` extendido (`src/hooks/useNoteSave.ts`):** `SaveStatus` extendido de 3 a 5 estados (`'idle' | 'saving' | 'saved' | 'retrying' | 'error'`). Eliminado `useState<SaveStatus>` local — status derivado enteramente del queue via `useSyncExternalStore`. `save()` callback refactor: en lugar de `try/await/catch + setStatus('idle')` silencioso, calcula payload + ejecuta `syncLinks` 1x, y delega persistencia a `saveContentQueue.enqueue(noteId, payload, (p) => notesRepo.saveContent(noteId, p))`. Guard único: si entry actual del queue está en `'error'`, NO encolar (el usuario debe resolver via banner). Latch del badge "✓ Guardado" 1.5s al transicionar a synced — el queue GC-ea la entry tras 100ms; el latch local mantiene el badge visible su ventana completa. Cleanup F22 mantenido (timers locales del debounce); NO se cancela el queue entry en unmount: el retry sigue corriendo background si el usuario navega a otra nota.

- **F28.3 — Banner UI con [Reintentar / Descartar / Copiar contenido]:** componente nuevo `src/components/editor/SaveErrorBanner.tsx`. Mounted en `NoteEditor.tsx:97-98` entre `<SummaryL3>` y `<DistillLevelBanner>`. Mismo wrapper canónico que DistillLevelBanner (`mx-auto w-full max-w-180 px-4 pt-3` + animaciones `animate-in fade-in slide-in-from-top-1`). Returns null si `status !== 'error'`. Acciones:

  - **Reintentar**: `saveContentQueue.retryNow(noteId)`. Reset attempts a 0, re-ejecuta inmediato; si vuelve a fallar, entra al ciclo de retry y eventualmente al banner de nuevo.
  - **Descartar cambios**: `saveContentQueue.cancel(noteId)` + `onDiscard()` callback al padre. `NoteDetailPage` mantiene state `discardCount` (monotonic increment); el handler bumpea el counter, que se pasa a `useNote` como nuevo param `version` y como sufijo en `key={`${noteId}-${discardCount}`}` del `<NoteEditor />`. Re-mount limpio + re-fetch de Firestore con `getDoc()` fresco.
  - **Copiar contenido**: `navigator.clipboard.writeText(`${title}\n\n${contentPlain}`)` desde el payload del queue entry + feedback inline en el botón ("✓ Copiado" 2s + disabled). Sin lib de toasts. El banner permanece tras copiar — escape puro de último recurso, no resolución.

  `SaveIndicator` extendido con 2 estados nuevos: `'retrying'` ("Reintentando..." amber) y `'error'` ("Error al guardar" destructive). `useNote.ts` modificado para aceptar param opcional `version: number = 0` agregado a las deps del useEffect.

- **F28.4 — Flush hooks (`src/hooks/useSaveQueueFlush.ts` + extensión de `useCloseToTray.ts`):** hook nuevo registra 2 listeners globales:

  - `'beforeunload'`: si hay queue entries pending, dispara `event.preventDefault()` + `event.returnValue = ''` para mostrar el dialog nativo "Hay cambios sin guardar" + intento best-effort `flushAll()` async. Sin entries, cierre limpio sin dialog.
  - `'online'`: cuando navigator vuelve online, `flushAll()` inmediato si hay entries. Recovery instantáneo sin esperar al próximo backoff tick.

  Mounted en `TauriIntegration.tsx` junto a `useCloseToTray` y `useAutoUpdate`. `useCloseToTray.ts` extendido: callback de `onCloseRequested` ahora es async; antes de `window.hide()`, race entre `flushAll()` y timeout 2s. Si flush succeeds: hide. Si timeout: hide igual (queue persiste in-memory con el proceso vivo en tray).

- **Hot-fix `savedBadgeVisible` (commit `6b19e36`):** detectado en QA E2E F28.5 — el badge "✓ Guardado" quedaba pegado en el SaveIndicator durante un nuevo ciclo de retry/error. Root cause: el `useEffect` con dep `[queueEntry?.status]` tenía el `setTimeout` y su cleanup conviviendo. Cuando la dep cambiaba (nuevo enqueue post-save exitoso), el cleanup ejecutaba `clearTimeout` cancelando el `setSavedBadgeVisible(false)` diferido, dejando el latch pegado en true. Fix: dividir en 2 effects siguiendo gotcha F22 — trigger effect (dep `[queueEntry?.status]`) que setea/resetea el badge, y auto-hide effect separado (dep `[savedBadgeVisible]`) con timeout 1.5s + cleanup propio. Match exacto al patrón canónico de `DistillLevelBanner.tsx`.

E2E validado en producción local con Playwright MCP:

- **Path feliz**: edición real → debounce 2s → SaveIndicator pasa por "✓ Guardado". Flow end-to-end con Firestore real.
- **Banner aparece tras 4 fallos**: enqueue forzado vía `import('/src/lib/saveQueue.ts')` desde la consola con executor que siempre rechaza. Tras ~7s (1+2+4 de backoff): banner visible con `role="alert"`, mensaje "No pudimos guardar tu última edición / Reintentamos 3 veces sin éxito...", 3 botones en orden Copiar/Descartar/Reintentar. SaveIndicator dice "Error al guardar".
- **[Reintentar]**: retryNow re-encola desde `attempts: 0`. Executor sigue rechazando → tras 7s vuelve a `'error'`.
- **[Descartar]**: queue vacío + banner oculto + editor recargado server-side ("Metodología SDD..." original).
- **[Copiar contenido]**: clipboard contiene `${title}\n\n${contentPlain}` correcto. Banner permanece visible (escape, no resolución).

## Lecciones

- **`useSyncExternalStore` + factory queue es un patrón canónico para state externo a React.** Match exacto con el patrón de `useOnlineStatus` (window events). El `subscribe(cb: () => void)` sin args + `getSnapshot()` separado es la API canónica que React espera. Aplicable a cualquier futuro store-like que no quepa en TinyBase: queues de side-effects, observable singletons, listeners globales con estado.

- **Inmutabilidad por entry, mutabilidad por Map.** React compara snapshots con `Object.is`. Si el `Map` se mantiene como referencia estable y solo cambia el contenido, `useSyncExternalStore` no re-renderea. La solución: cada `setStatus(id, status)` crea **nuevo objeto** (`entries.set(id, {...prev, status})`) — el `getEntry(id)` retorna referencia distinta si hubo cambio real. El Map mantiene identidad porque el cambio se observa per-id, no global.

- **Race cancel-vs-async-await sin AbortController.** No se puede abortar un `await updateDoc` imperativamente (no acepta AbortSignal). El patrón es: flag mutable en la entry + check antes de cualquier `setStatus` post-await. Match con cómo se hace cancellation en código async sin abort signal: re-leer el state externo después del await y validar que sigue siendo relevante. Aplicable a cualquier futuro queue con executores async.

- **Version-tagging resuelve "upsert durante in-flight" sin ser un compare-and-set explícito.** Cada `enqueue`/`retryNow` bumpea un counter global; el executor captura `startVersion` al inicio y compara al terminar. Si difiere, re-ejecuta. Patrón aplicable a cualquier queue donde el último-write debe ganar pero los awaits anteriores ya están en vuelo.

- **`useEffect` con `setTimeout(...)` + cleanup `clearTimeout(...)` en el mismo effect = badge pegado.** Cuando la dep cambia, el cleanup cancela el timer ANTES de que el `setState(false)` diferido corra → el latch queda pegado en `true` para siempre. Match con gotcha F22 ("auto-dismiss timer en useEffect separado del trigger"); F28 hot-fix confirma que el patrón aplica también para latches de status (no solo banners visuales). Regla: cualquier `setTimeout` que cierra un latch debe vivir en useEffect separado con dep `[latchValue]` — el trigger setea, el cleanup limpia el latch del trigger anterior y deja correr el del nuevo.

- **Upsert semantics importan más de lo que parecen.** El enqueue durante un retry pendiente no debe resetear attempts a 0 — la próxima vez que falle debe seguir consumiendo el backoff pendiente. Si reseteas, el usuario que tipea continuamente durante una caída de red queda en un loop infinito de retry @ 1s, en vez de avanzar 1s/2s/4s/error. La excepción es desde `'error'`: ahí sí se resetea, porque conceptualmente es equivalente a `retryNow()` con payload nuevo.

- **Detección de error permanente NO debe ser `instanceof Error`.** Solo `FirebaseError` con codes específicos. Un `new Error('quota exceeded')` plano genérico no debe entrar al fast-fail — puede ser cualquier cosa. La regla del code es semántica: `permission-denied` significa `auth.currentUser.uid` no tiene permiso (token rotado, rule cambiada, etc.); `failed-precondition` típicamente significa transaction conflict; `invalid-argument` típicamente payload >1MB o serialización rota; `unauthenticated` token expired. Ninguno se resuelve con tiempo.

- **Discard via key bump es más limpio que rollback con commands de TipTap.** Reaprovecha el ciclo natural unmount→mount del componente. El `useNote` con dep `[user, noteId, version]` re-corre el `getDoc` desde cero, parseando el `content` JSON server-side fresh. No hay que mantener un snapshot del content pre-edit ni manipular el editor instance — la alternativa "elegante" es la más simple aquí.

- **QA E2E forzando estado del queue desde la consola es viable en dev.** Vite expone módulos del proyecto via `import('/src/...')` desde `browser_evaluate`. Un singleton exportado del módulo es la MISMA instancia que la app está usando — un `enqueue` desde la consola triggea el `useSyncExternalStore` real del componente. Permite testing visual de estados raros (4 fallos consecutivos transient) sin necesidad de network mocking complejo. Limitación: imports cross-package (`firebase/app` para construir un FirebaseError) NO resuelven desde el evaluate — usar Error nativo y aceptar el ciclo completo de retries, o agregar un helper temporal en window. Aplicable a futuros queues, observables, o servicios con singleton patterns.

## Out of scope (deuda explícita, escalada a ESTADO-ACTUAL)

- **Retry queue extendido al factory `createFirestoreRepo` para `updateMeta`, `softDelete`, `acceptSuggestion`, `toggleFavorite`, `createFromInbox`, etc.** El patrón roto (sync TinyBase + async Firestore + catch silente) vive en TODO write del factory. F28 lo cierra solo en `saveContent` por priority (alta velocidad + contenido del usuario). Los demás writes son metadata baja-velocidad: si fallan, el usuario ve que el toggle no aplicó y reintenta. SPEC futuro F29+ debe llevar el patrón al factory entero, reusando la primitiva `createSaveQueue` con tipo genérico por entidad (notes/tasks/projects/etc.).

- **Activación de Firebase IndexedDB persistence.** Decisión separada con análisis de duplicación con TinyBase persister. Cubriría el 80% del caso (offline transient) gratis sin app-layer retry. NO bloqueante: el queue F28 cubre los casos transient en la sesión actual. Si los logs en producción muestran que los retries cross-session (cerrar tab + volver con red restaurada) son frecuentes, evaluar.

- **Persistencia del queue en IndexedDB propio.** Sobrevive reload/crash. Trivial extensión del `createSaveQueue` con storage adapter. NO bloqueante por la misma razón anterior.

- **Telemetría / observabilidad de retries en producción.** Cuántos saves fallaron en N días, distribución de attempts, error codes. Útil para dimensionar si IndexedDB persistence vale la complejidad. Fácil de agregar — `onAttempt`/`onError` callbacks en el factory + analytics.

- **Subscription granular per-noteId en `useSyncExternalStore`.** El patrón actual notifica TODAS las suscripciones cuando muta el Map por cualquier noteId. Re-renderea editores de notas no afectadas si hay >1 entry simultánea en queue. No crítico para v1 (caso típico es 1 nota activa con 0-1 entry pending). Si aparece como bottleneck, refactor a `subscribe(noteId, cb)` con notify selectivo.

- **`setPartialRow` stale en TinyBase + `syncLinks` rollback path.** Cuando el usuario "Descarta" tras error, el `setPartialRow` ya escribió título/preview/metadata a TinyBase y el persister ya pudo haber sincronizado a Firestore. El editor revierte al content server (correcto) pero la metadata visible en lista (preview/título) puede haber avanzado. El banner deja claro: "Descartar revierte el título visible" — pero la solución técnica completa es coordinar el rollback de TinyBase + cleanup de `syncLinks`. Requiere rewrite del factory. Deuda visible para SPEC futuro.
