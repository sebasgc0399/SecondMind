## Objetivo

El widget FSRS hoy comparte espacio con hubs en `DailyDigest` — el user no ve un "te tocan N hoy" ni un acceso a la lista completa cuando hay más due que las 3 que el digest muestra. F24 separa el resurfacing en su propia card con copy diferenciado (`Nada que repasar` / `Te toca 1` / `Te tocan N`), agrega "Ver todas las due" cuando supera 5, y abre una tab "Por revisar" en `/notes` para revisión deliberada. La feature más usada del user gana prominencia y deja de pelear espacio con la lista de hubs.

## Lecciones

- **`useState` lazy initializer leyendo `useSearchParams` para deep-link unidireccional URL→state.** Más simple que `initRef + useEffect`, sin flicker, sin re-trigger ambiguo cuando react-router re-instancia el objeto. Patrón canónico: `const [filter, setFilter] = useState<Filter>(() => { const f = searchParams.get('filter'); return f === 'review' ? 'review' : 'all'; })`. Aplicable a cualquier futura page que necesite read-once URL params al mount.
- **Reusar mappers genéricos (`rowToOramaDoc`) en hooks que devuelven items para `<NoteCard>`.** Evita la trampa de extender helpers acoplados a contextos específicos (`trashNoteToOramaDoc` con `aiSummary: ''` hardcoded y `daysUntilPurge`). El patrón "hook devuelve `BaseShape & { extraField }` y deconstructura `{ extraField, ...base }` en el render" funciona via structural subtyping de TypeScript sin helpers de mapeo extra. Aplicable a futuras vistas filtradas que reusen el mismo NoteCard.
- **Sub-split de commits cuando dos features tocan el mismo archivo.** Cuando un refactor (rename/move) precede a una expansión (agregar item nuevo) y ambos tocan el mismo archivo, splittear: refactor hace sustitución 1:1 in-place, expansión hace la reorganización completa. Beneficio: evita conflicts en auto-merge, cada commit deja main shippable, history más navegable. Aplicable a cualquier secuencia rename + add en el mismo archivo.
