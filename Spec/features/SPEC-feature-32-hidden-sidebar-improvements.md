# SPEC F32 — Hidden Sidebar Mode Improvements (Registro de implementación)

> Estado: Completada Abril 2026
> Commits: `83c0985` SPEC, `7656f9f` F32.2 event.code, `8332934` F32.4 hint anti-flash, `425dce5` F32.1 toggle palette, `da7cf1c` F32.3 animación toggle-only.
> Gotchas operativos vigentes → `Spec/ESTADO-ACTUAL.md` y `CLAUDE.md`

## Objetivo

Cerrar las cuatro deudas explícitas del out-of-scope de F31 sin alterar el modelo persistente de `UserPreferences` ni las decisiones D1–D8 ya consolidadas. Resultado: toggle bidireccional desde Command Palette equivalente al shortcut Cmd+B, layout-independence universal de los dos shortcuts legacy del repo (Cmd+K y Alt+N) migrados al patrón `event.code` canonizado en F31.5, animación visual del swap sidebar↔TopBar disparada solo por interacción (no por mount inicial post-anti-flash), y eliminación del flash sidebar→TopBar de ~100-300ms para usuarios con `sidebarHidden=true` persistido. Cero cambios en schema Firestore, security rules ni Cloud Functions; cero migración de datos.

## Qué se implementó

- **F32.2 — Migración Cmd+K y Alt+N a `event.code` (`src/components/layout/CommandPalette.tsx`, `src/components/capture/QuickCaptureProvider.tsx`):** los dos shortcuts existentes del repo dejan de matchear con `event.key` y pasan a `event.code === 'KeyK'` / `'KeyN'`. `CommandPaletteProvider` suma checks explícitos `!event.shiftKey && !event.altKey` para paridad con el patrón canónico de `useSidebarVisibilityShortcut`. `QuickCaptureProvider` simplifica el doble check `'n'/'N'` a un solo `event.code === 'KeyN'` (event.code no varía con shift). Sin guards de `activeElement` agregados — F32 cierra layout-independence, no foco-awareness; Cmd+K/Alt+N no colisionan con atajos del editor TipTap. Comentario explicativo apunta a F31.5 como precedente.

- **F32.4 — localStorage hint anti-flash para `sidebarHidden` (`src/lib/preferences.ts`, `src/hooks/usePreferences.ts`, `src/app/layout.tsx`):** helpers nuevos `readSidebarHiddenHint(uid: string): boolean | null` y `writeSidebarHiddenHint(uid: string, value: boolean): void` con key `'secondmind:sidebarHidden:' + uid` y try/catch defensivo. `usePreferences` hidrata `sidebarHidden` síncronamente desde el hint en el effect de `[user]` antes de subscribir; en el callback de `subscribePreferences`, escribe el hint solo cuando `loaded === true` (evita pisar hint válido con DEFAULT pre-snapshot). `layout.tsx` quita el AND-gate `prefsLoaded && preferences.sidebarHidden` introducido en F31 (D7 anti-flash) — ahora `sidebarHiddenEffective = preferences.sidebarHidden` directo, porque el hint llena el gap pre-isLoaded. Otros campos de `UserPreferences` (`distillIntroSeen`, `distillBannersSeen`, `trashAutoPurgeDays`) siguen gobernados por `isLoaded` gate dentro de sus consumers — disparan side-effects que no pueden actuar contra hint stale. Hint key incluye uid para no contaminar entre cuentas en multi-account browsers.

- **F32.1 — Toggle bidireccional desde Command Palette (`src/components/layout/CommandPalette.tsx`):** `QuickAction` pasa de interface plano a discriminated union `{ kind: 'navigate' | 'handler' }`. Las 8 entries existentes mantienen `'navigate'` con `url`; una novena entry dinámica de toggle se agrega solo cuando `breakpoint === 'desktop' && user`. Estado `sidebarHidden=false` → entry "Ocultar sidebar" con icono `PanelLeftClose`; estado `sidebarHidden=true` → "Mostrar sidebar" con `PanelLeftOpen`. `quickActions` se computa con `useMemo` deps `[breakpoint, user, preferences.sidebarHidden]` — la memo se invalida cuando el state cambia, así el label queda en sync si Cmd+B o /settings disparan el toggle mientras el palette está abierto. `navigateTo` branchea por kind: `'result'` → navigate URL del search result, `'navigate'` → navigate URL de quick action, `'handler'` → invoca handler fire-and-forget (`void setPreferences(uid, { sidebarHidden: !current })`). `getItemUrl` se refactorizó a `getResultUrl` (solo SearchResult). Tablet 768 y mobile 375 NO reciben la entry — el array original de 8 se devuelve tal cual. `// eslint-disable-next-line import/order` sobre el import de `@/lib/preferences` mitiga un quirk del plugin con inline type imports en `@/hooks/*` arriba (mismo patrón que la directiva preexistente sobre `@/hooks/useGlobalSearch`).

- **F32.3 — Animación toggle-only sidebar↔TopBar (`src/app/layout.tsx`, `src/components/layout/Sidebar.tsx`, `src/components/layout/TopBar.tsx`):** Sidebar y TopBar suman prop opcional `animateEntry?: boolean`; cuando es `true`, el componente recibe `animate-in slide-in-from-left duration-200` (sidebar) o `animate-in slide-in-from-top duration-200` (TopBar) vía `cn()`. Layout coordina con `useRef<boolean>(true)` para detectar mount inicial, `useState<boolean>(false)` para `animateLayoutSwap`, y un `useLayoutEffect` con dep `[preferences.sidebarHidden]` que sale temprano en el primer render y dispara `setAnimateLayoutSwap(true) + window.setTimeout(false, 300)` con cleanup en cambios subsecuentes. Se eligió `useLayoutEffect` (no `useEffect`) porque el setState debe correr ANTES del paint para que el componente entrante reciba la clase animate-in en su PRIMER frame visible — con `useEffect` se descubrió en E2E un blip de ~30ms donde el elemento aparece en posición final y luego retro-anima. El componente que se desmonta sale instantáneo: approach asimétrico intencional, 80% del valor visual sin la complejidad de retardar unmount con AnimatePresence custom.

E2E validado en sesión de cierre con Playwright MCP en viewports `1280×800` (desktop), `768×1024` (tablet), `375×667` (mobile):

- Desktop default: sidebar visible, sin TopBar.
- Cmd+K en desktop → palette con "Ocultar sidebar" como 9ª entry (icon `PanelLeftClose`); Enter oculta sidebar + monta TopBar + cierra palette.
- Cmd+K en hidden mode → palette muestra "Mostrar sidebar" reactivamente (icon `PanelLeftOpen`); Enter restaura sidebar.
- Cmd+B desde sidebar visible → TopBar entra con `animate-in slide-in-from-top duration-200` desde el primer frame post-mount; sampling DOM cada 30ms confirmó la clase aplicada en el frame visible inicial y removida tras 300ms.
- Cmd+B desde TopBar → sidebar entra con `animate-in slide-in-from-left duration-200` simétricamente.
- Reload con `localStorage.sidebarHidden = 'true'` → TopBar visible desde frame 1 sin slide-in (mount inicial NO anima); console limpia (0 errors).
- Tablet 768 + Mobile 375 → Cmd+K muestra 8 entries originales (sin la entry de toggle).
- Alt+N en desktop → QuickCapture dialog abre como esperado.
- Toggle write actualiza `localStorage.sidebarHidden:gYPP7NIo5JanxIbPqMe6nC3SQfE3` sincrónicamente al snapshot.
- Console limpia tras todo el flujo (0 errors después del reload final).

## Lecciones

- **`useLayoutEffect` (no `useEffect`) cuando el setState ES la fuente que activa una clase de animación pre-paint.** Cualquier componente que monta tras un cambio reactivo (toggle de prop, snapshot Firestore, breakpoint cross-thresh) y recibe una clase `animate-in` condicional vía state debe disparar ese setState ANTES del paint. Con `useEffect` se ve el componente en posición final ~30ms y luego retro-anima — blip visual perceptible aunque sutil. `useLayoutEffect` resuelve causando un re-render sync pre-paint. El costo es trivial (un setState extra cuyo efecto es inmediato), y es la fix idiomática para este escenario. Patrón aplicable a cualquier futura swap-anim de chrome global (sidebar/TopBar/header alterno, tabs swipe, drawer slide, modal entry) que necesite distinguir mount inicial de toggle interactivo. Escalado a `CLAUDE.md` como gotcha universal porque el patrón aplica a toda sesión sin importar el dominio.

- **Hint anti-flash en localStorage es la respuesta correcta para flags solo-UI con persistencia Firestore que afectan layout.** El AND-gate de F31 (D7) era una mitigación honesta pero conservadora — defaulteaba a "sidebar visible" pre-snapshot, generando flash inverso para usuarios con `hidden=true` persistido. F32.4 demuestra que cachear el último valor en localStorage por uid permite hidratación síncrona pre-primer-onSnapshot. Limitación clave: solo aplica a flags que afectan layout sin disparar side-effects (banners one-time, intros, etc. NO califican porque ejecutarían contra hint stale). Aplicable a futuras prefs como tema custom, modo focus, layout density, etc. cuando aparezcan. Patrón canónico vivo en `src/lib/preferences.ts` (`readSidebarHiddenHint`/`writeSidebarHiddenHint`) y `src/hooks/usePreferences.ts` (hidratación pre-subscribe + write-on-snapshot con guard `loaded === true`).

- **Discriminated union sobre interface plano cuando una entidad gana variantes con distinto contrato.** `QuickAction` empezó en F31 como `{ id, label, icon, url }` — un solo flujo navigate. F32.1 introduce un segundo flujo (handler fire-and-forget) que NO tiene `url`. El cambio mecánico de "agregar handler? opcional" rompe el contrato (un consumer que asume `url` always defined falla en runtime). Discriminated union con `kind: 'navigate' | 'handler'` fuerza al type system a branchear: `navigateTo` no compila si olvidás un caso. Aplicable a cualquier entidad domain que crezca en variantes con shapes distintos — tarjetas dashboard, results de búsqueda con tipos heterogéneos, items de inbox post-IA, etc.

- **E2E sampling cada N ms detecta blips visuales que un screenshot post-acción enmascararía.** Un solo snapshot 100ms post-toggle hubiera mostrado la animación ya en curso (clase aplicada). Sampling cada 30ms reveló el gap de 30ms donde el componente está mounted SIN la clase — diagnóstico que motivó el cambio a `useLayoutEffect`. Patrón aplicable a cualquier feature de animación reactiva: instrumentar con `mcp__playwright__browser_evaluate` que hace `setTimeout(snap, N) × M` post-trigger, devolver array de samples, leer evolución temporal. Más confiable que screenshot single-frame y más barato que video recording.

- **El quirk de `eslint-plugin-import` con inline type imports persiste.** F31 tenía un `// eslint-disable-next-line import/order` para `@/hooks/useGlobalSearch` por inline `{ type SearchResult }`. Al re-ordenar imports en F32.1, el quirk se manifestó en otra línea (`@/lib/preferences`). El plugin reclasifica imports con inline type keyword en un grupo distinto, y `newlines-between: never` dispara newlines-between phantom entre groups que no deberían serlo. Workaround estable: `eslint-disable-next-line import/order` en la línea ofensora con comentario explicativo. Si futuras features tocan imports en archivos con inline type imports, anticipar el quirk.

## Out of scope (deuda explícita)

- **AnimatePresence-style exit-anim para que el saliente también deslize fuera.** F32.3 anima solo el entrante; el saliente desmonta instantáneo. Si el usuario reporta que la asimetría se siente incompleta, escalar a un wrapper que retarde unmount con `setTimeout + state isExiting`. Approach descartado en F32 por costo medio (refactor de mount logic) sobre 20% del valor visual.
- **`getModKey()` util cross-platform** para renderizar `⌘K` en mac y `Ctrl K` en win/linux dinámicamente. Hoy hardcodeamos `⌘K` en TopBar y `Alt+N` en QuickCaptureButton. Follow-up si llega feedback de UX. Ya estaba documentado en F31.
- **Cleanup explícito del hint en signOut.** Hoy el hint persiste post-signOut; el próximo login del mismo uid hidrata correcto, y de otro uid usa su propia key. No es leak de privacidad (dato boolean trivial), pero un sign-out "limpio" podría purgar todos los `secondmind:*` keys. Decisión: aceptable como está; revisitar si surgen otros hints.
- **Refactor a `useQuickActions()` extraído** si el array crece más allá de 9-10 entries con varias dinámicas. Hoy 9 entries con 1 dinámica viven cómodas en `useMemo` inline.
- **Race del hint stale cross-device** sigue siendo aceptable v1 (documentado en checklist E2E). Usuario con device A en `hidden=true` cambia a `false` desde device B, abre device A → hint dice `true` → flash inverso TopBar→Sidebar de una vez al llegar el snapshot. Mitigación futura: invalidar hint en sign-in si el snapshot diverge — pero genera otro flash.
