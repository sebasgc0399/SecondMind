## Objetivo

Las novedades de cada versión solo se veían **una vez**, en el modal what's-new que dispara tras actualizar. Quien lo descartaba (o saltaba varias versiones de un update) no podía volver a verlas: el catálogo `CHANGELOG_ENTRIES` existía en el bundle pero ninguna UI lo listaba. La feature agrega una **sub-página** (`/settings/changelog`, enlazada desde `AppInfoSection`) que lista **todas las entradas de changelog liberadas**, más nueva primero, bilingüe, en los 3 frentes — con **cero estado nuevo**: resuelve (a) re-ver novedades on-demand y (b) catch-up de versiones saltadas. Single-concern: mostrar el catálogo existente; no toca el trigger del modal, ni `lastSeenVersion`, ni introduce comparación semver.

**Diferido (OUT, con razón):** badge "no-leído" + highlight de entradas posteriores a `lastSeenVersion` (requieren estado `has-unread` persistido nuevo + lógica de lectura — YAGNI, se reconsidera con evidencia); accordion/paginación (catálogo chico); campo `date` por entrada (aditivo); re-trigger del modal; `semver.compare()`.

## Lecciones

- **`i18next-cli types` genera el `.d.ts` en comillas dobles y corre por shell → el hook Prettier no se dispara.** Tras regenerar tipos hay que `npx prettier --write src/types/resources.d.ts`, o el diff mete ~1000 líneas de ruido de formato (pasó en F3, corregido con `--amend`). Generaliza a cualquier artefacto regenerado por un CLI fuera del flujo Edit/Write del harness. Escalado a `gotchas/i18n.md` + Paso 2.5 de `release-ecosystem`.
- **El comando de regen es `i18next-cli types` (lee los JSON), no `extract` (escanea el código y purga con `removeUnusedKeys`).** Para agregar keys: editar los JSON + `types`. Nunca `extract` acotado por scope.
- **Un test que usa un fixture compartido como fuente REAL se rompe al mutar el fixture — preempt-earlo.** El caso 3 de `useWhatsNew.test.ts` no mockeaba `@/lib/changelog` (usaba el registry real `0.6.0`→`v060`); removida `v060`, el modal dejaba de abrir. Antes de tocar un registry/fixture, grepear qué tests lo consumen sin mock y reescribirlos en el mismo commit.
- **Testear orden reverse con un catálogo de 1 entrada: mockear el registry con ≥2 entradas + atributo `data-version`.** Con 1 entrada el reverse no se distingue; mockear `CHANGELOG_ENTRIES` con 2 (key i18n real ×2) y afirmar el orden por `data-version` (no por título, que comparten) lo cubre sin inventar keys i18n que `tEs` rechazaría.
- **Con worktrees paralelos, `main` avanza bajo los pies — re-verificar el auto-merge.** Durante F60, el trabajo paralelo mergeó F61 (renumerada 60→61 por colisión de nombre); el `--no-ff` auto-fusionó `ESTADO-ACTUAL.md` sin conflicto. Un auto-merge "exitoso" puede combinar mal → confirmar por grep que ambas features sobreviven + re-correr el gate en `main` post-merge.
