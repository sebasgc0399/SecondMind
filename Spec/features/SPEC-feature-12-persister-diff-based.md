# SPEC — Feature 12: Persister diff-based (Registro de implementación)

> Estado: Completada Abril 2026
> Commits: `c97ab20` SPEC refinado post-auditoría, `e3cb657` F1+F2 refactor `setPersisted` con `changes` nativo + `Promise.allSettled` + `onIgnoredError`, `833733c` 8 tests Vitest + ajuste por limitación TinyBase v8 (deletes no reportados).
> Gotchas operativos vigentes → `Spec/ESTADO-ACTUAL.md`

## Objetivo

El callback `setPersisted` del persister TinyBase ↔ Firestore iteraba todas las rows de la tabla en cada save. Con 500 items y 1 update, disparaba 500 setDocs redundantes. F12 delega el diff a la API nativa `changes` (param 2 de `setPersisted` en TinyBase v8) — el callback recibe el delta de la transacción ya computado y emite `setDoc(merge:true)` solo para las rows tocadas. Reduce write amplification de O(N) a O(cambios) sin mantener ningún snapshot manual.

## Qué se implementó

- **F1 — Refactor `setPersisted` con `changes` nativo:** el callback ahora consume `(getContent, changes?) => ...` directamente. Itera `changes[0][tableName]` y emite `setDoc(merge:true)` para cada row reconstruida desde `getContent()`. `Promise.allSettled` (no `Promise.all`) para que un setDoc fallido no aborte los paralelos del mismo tick. Rejects recolectados al final, filtrados con helper `isSilentError` (filtro `SILENT_ERROR_CODES` generalizado a writes), re-throwed como `AggregateError` si múltiples para que `onIgnoredError` (6º arg de `createCustomPersister`) los reciba. TinyBase NO reintenta automáticamente — rows fallidas quedan eventualmente consistentes solo cuando vuelven a tocarse. Archivos tocados: [src/lib/tinybase.ts](../../src/lib/tinybase.ts).
- **F2 — Fallback `changes === undefined`:** skip + `console.debug` en dev. Cubre el caso típico del primer tick post-`startAutoLoad` (sin cambios pendientes, solo carga). Validado empíricamente con log temporal: las 7 tablas del proyecto invocan `setPersisted` con `changes === undefined` en el primer tick; tras una mutación real (`store.setRow` por toggle de hábito), `changes` llega definido como `Array(3)` solo para la tabla afectada. El comportamiento conservador "escribir todo si no sé qué cambió" anularía el ahorro en el caso dominante (load) → skip explícito. Archivos tocados: [src/lib/tinybase.ts](../../src/lib/tinybase.ts).
- **F3 — 8 tests Vitest del persister:** patrón de mocks reusado de [baseRepo.test.ts](../../src/infra/repos/baseRepo.test.ts) — `vi.mock('firebase/firestore')` + `vi.mock('@/lib/firebase')` + `Store` real construido con `createStore().setTablesSchema(...)`. Tests 1, 2, 3, 5, 7 cubren el path principal (skip post-load, setRow, setPartialRow row entero, transacción x3, fallo parcial sin retry con `vi.spyOn(console, 'error')`). Tests 4, 6, 8 documentan la limitación TinyBase v8 con asserts negativos (delRow/delTable no disparan deleteDocs). 13/13 tests pasan en suite completa (5 baseRepo + 8 persister). Archivos tocados: [src/lib/tinybase.test.ts](../../src/lib/tinybase.test.ts).
- **F4 — Validación E2E + métricas:** ejecutada con Playwright MCP + Firebase MCP en dev local. Carga inicial: 7 logs `setPersisted sin changes — skip` (uno por tabla), 0 errors, 0 warnings — loop load→write evitado. Toggle de hábito ("Ejercicio"): 0 mensajes adicionales en consola, persistencia confirmada en Firestore vía `firestore_get_document` (`ejercicio: true`, `progress: 7`, `updatedAt: 1776832710696`, los 13 hábitos restantes preservados como `false` → `merge: true` funciona). Escenarios A (write amplification con N=50) y C (cross-user real) diferidos como dogfooding — el comportamiento del callback ya está cubierto por tests unitarios.

## Decisiones clave

| #   | Decisión                                                                                            | Razón                                                                                                                                                                                                                                                                                                                                                    |
| --- | --------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| D1  | Delegar diff a la API nativa `changes` de TinyBase v8 (no mantener `previousRows` snapshot manual). | Auditoría reveló que `PersistedChanges<StoreOnly> = Changes` ya trae el delta computado. El snapshot manual era reimplementación de una API existente: ~40 LoC innecesarias (helpers clone + shallow equal), closure con estado que sincronizar, tests difíciles de escribir (`previousRows` no observable sin hacks).                                   |
| D2  | Granularidad del setDoc: row entera con `merge:true`, no cells parciales.                           | Reconstruir un payload parcial desde cell-changes (`{cellId: undefined}` = deleted) requiere más código y los rows son ≤20 campos pequeños. Con `merge:true` garantizado, escribir el row entero pesa lo mismo en bytes. El ahorro real de F12 viene del nivel de "qué rows se escriben", no de "qué campos por row".                                    |
| D3  | `Promise.allSettled` para writes paralelos del mismo tick (no `Promise.all`).                       | Con `Promise.all`, el primer reject hace que el `await` retorne sin esperar las demás promesas (que siguen en vuelo, sin observación). Un `setDoc` fallido abortaría otros OK del mismo tick. Con `allSettled` cada write completa independiente y reportamos rejects juntos via `AggregateError`. Costo: cero (TinyBase no reintenta igual — ver D4).   |
| D4  | No retry automático ante fallo de `setPersisted` (eventual consistency).                            | Auditoría confirmó que TinyBase no reintenta cuando el callback rechaza — solo llama a `onIgnoredError`. El siguiente save trae solo las rows cambiadas desde el último save exitoso. Implementar retry custom duplica lógica del SDK y arriesga loops en logout legítimo. Las rows fallidas quedan eventualmente consistentes cuando vuelven a tocarse. |

## Lecciones

- **Antes de implementar un mecanismo, verificar si la API base ya lo provee nativo.** El SPEC original mantenía `previousRows` snapshot + helpers `cloneRows`/`shallowEqualRow` para diff manual. La auditoría reveló que TinyBase v8 entrega `changes` como 2º arg de `setPersisted` desde v4.0+. Eliminar ~40 LoC innecesarias. Generalizable: cualquier custom persister, custom hook con state derivado, o wrapper sobre librería establecida — leer los `.d.ts` antes de reimplementar.
- **`null` vs `undefined` en types vs JSDoc puede divergir.** El TYPE de `Changes` ([store/index.d.ts:1521-1529](../../node_modules/tinybase/%40types/store/index.d.ts#L1521-L1529)) permite solo `undefined` para "row eliminada", pero el JSDoc del mismo type usa `null` como ejemplo (`"fido": null`). Defensivo: comparar con `== null` (doble igual) cubre ambos sin costo de runtime. Aplica a cualquier API donde JSDoc y TYPE se actualizan de forma desincronizada.
- **`Promise.allSettled` vs `Promise.all` cuando los writes deben ser independientes.** Con `Promise.all`, un primer reject termina el `await` sin esperar las demás promesas — siguen en vuelo pero el caller no las observa. Para batches de writes paralelos donde un fallo no debe abortar los demás (caso típico: persisters, fan-out de notificaciones, batch operations), usar `allSettled` y recolectar rejects al final con `AggregateError` para reportar juntos.
- **TinyBase v8 NO incluye row IDs eliminados en `changes`.** `delRow` standalone produce `changes = [{}, {}, 1]` (vacío); `delTable` igual. Limitación descubierta empíricamente con un debug test que inspecciona el shape real del param. El gotcha original de F11 ("delTable con persister snapshot-based no borra Firestore") sigue siendo cierto post-F12 pero por una razón distinta: TinyBase no reporta deletes (vs el persister no diff los elimina). Patrón de auditoría reusable: cuando un test falla por comportamiento inesperado de una API externa, escribir un test mínimo que inspeccione el shape exacto antes de adaptar el código — así el fix se hace sobre evidencia, no sobre hipótesis.
- **Validación empírica pre-código como gate explícito en el plan.** R2 del plan (shape de `changes` durante el primer tick post-`startAutoLoad`) podía romper la decisión de skip + log. Antes de codear el callback final, levantamos un dev server con un log temporal en el persister actual + abrimos la app + ejercitamos un par de mutaciones para confirmar el shape real. Costo: 5 min. Beneficio: si la asunción fallaba, pivotábamos al snapshot híbrido sin escribir + revertir código. Generalizable: cuando un plan depende de un comportamiento empírico no documentado claramente, gate explícito de validación ANTES del refactor + revert.
- **`onIgnoredError` (6º arg opcional de `createCustomPersister`) cubre rejects del callback.** Sin este hook, los `throw` de `setPersisted` quedan completamente silenciosos en producción. TinyBase no reintenta — solo entrega el error a este callback. Para cualquier custom persister conectarlo SIEMPRE con un logger que respete el filtro de errores benignos del proyecto (en este caso `SILENT_ERROR_CODES` para `permission-denied` / `unauthenticated` durante logout).
- **Plan mode + 3 Explore agents en paralelo + 1 Plan agent es alto leverage.** La auditoría descubrió 3 cosas que invalidaban parte del SPEC original: (1) `changes` nativo, (2) no retry automático, (3) `serverTimestamp()` como riesgo teórico no real. El SPEC v2 quedó simplificado a 4 features (vs 6 original) y los tests cambiaron de "observar closure" a "observar mocks". Aplicable a cualquier feature donde el SPEC depende de asunciones sobre API de librería externa o framework — la auditoría previa cuesta 30 min y ahorra horas de implementación equivocada.
