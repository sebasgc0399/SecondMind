> Gotchas operativos vigentes → `Spec/ESTADO-ACTUAL.md`

## Objetivo

Cerrar el silent data loss en TODOS los writes del factory `createFirestoreRepo` (`update`/`remove`) y los bypasses con `arrayUnion` (`notesRepo.acceptSuggestion`, `notesRepo.dismissSuggestion`). Pre-F29: el patrón roto vivía en TODO write metadata (toggle favorito, soft-delete, toggle hábito, `updateTask({ noteIds })`, `markProcessed`, `acceptSuggestion`) — `setPartialRow` mutaba TinyBase sync, `await setDoc` async fallaba por red, el `try/catch` tragaba el error con `console.error`, y al recargar `onSnapshot` pisaba el optimistic con server stale. F28 lo cerró solo en `saveContent` por priority (alta velocidad + contenido del usuario). F29 lleva el patrón al factory entero, reusando la primitiva `createSaveQueue<T>` con singletons tipados por entidad.

Solución: extender `createFirestoreRepo` con un `queue?: SaveQueue<Partial<Row>>` inyectable. `update`/`remove` delegan al queue cuando existe, con misma semántica que `saveContentQueue` (backoff `[1s, 2s, 4s]` × 4 attempts, fast-fail FirebaseError permanent, upsert sin reset de attempts, race cancel-vs-await, LRU 50). Los bypasses con `arrayUnion` reusan `saveNotesMetaQueue` con keys compuestas (`${noteId}:accept-${suggestionId}`). Sin queue → comportamiento actual (retro-compatible).

UI: `<PendingSyncIndicator />` en sidebar (desktop) + header (mobile), suscrito a los 7 queues vía agregador `usePendingSyncCount`. Estados: oculto si total = 0, "N pendientes" amber si todas en pending/syncing/retrying, "N sin guardar" destructive si ≥1 en error. Popover muestra lista per-entidad con plural correcto + acciones [Reintentar / Descartar].

## Qué se implementó

- **F29.1 — Singletons + `clear()` + `repoRows.ts` (`src/lib/saveQueue.ts`, `src/types/repoRows.ts`):** 6 nuevos singletons al final de `saveQueue.ts`: `saveNotesMetaQueue`, `saveTasksQueue`, `saveProjectsQueue`, `saveObjectivesQueue`, `saveHabitsQueue`, `saveInboxQueue`. Tipados como `SaveQueue<Partial<NoteRow>>`, etc. Arrays exportados `metaQueues` (los 6) y `allQueues` (`[saveContentQueue, ...metaQueues]`) para iteración. Tipos `NoteRow/TaskRow/ProjectRow/ObjectiveRow/HabitRow/InboxRow` extraídos a leaf nuevo `src/types/repoRows.ts` para evitar ciclo compile-time (saveQueue.ts → repoRows.ts ← \*Repo.ts via `import type`). Método nuevo `clear(): void` en la interface `SaveQueue<T>` y en el factory: itera entries, `clearTimeout` de timers pending, `entries.clear()`, `notify()`. NO setea `disposed=true` (vs `dispose()` que silencia futuros enqueues). Doble uso: `beforeEach` tests entre suites + acción "Descartar todo" del indicator. Tests vitest +1 escenario (15 total): `clear() vacía entries y notifica, mantiene subscribers vivos`.

- **F29.2 — Factory `createFirestoreRepo` con `queue?` + `removeRaw` (`src/infra/repos/baseRepo.ts`, `baseRepo.test.ts`):** `RepoConfig<Row extends RepoRow>` parametrizado con `queue?: SaveQueue<Partial<Row>>`. Body de `update`: `setPartialRow` sync first → si hay queue, `enqueue` con executor que recheca uid + llama `setDoc(merge:true)` → si no, await directo. Body de `remove`: análogo con `delRow` + `deleteDoc`. Método nuevo `removeRaw(id)` SIEMPRE bypassa queue (`delRow` + `await deleteDoc`) — necesario para `purgeAll` que usa `Promise.allSettled` sobre 50 paralelos: encolar 50 hits LRU cap (50). Sign-out mid-retry guard: el executor recheca `auth.currentUser?.uid !== capturedUid` al inicio de cada attempt. Si difiere, throw `Error('[repo] uid changed mid-retry')` (NO FirebaseError → no permanent → 4 attempts → 'error' visible en indicator). Mitiga el silent cross-user write sin agregar API surface. Tests `baseRepo.test.ts` +4: update con queue → enqueue + executor llama setDoc; remove con queue → enqueue + executor llama deleteDoc; removeRaw bypassa queue (deleteDoc directo, no enqueue); update con queue + uid distinto en retry → executor throws stale-write.

- **F29.3 — Wiring per repo (single commit, mecánico):** 6 archivos de repo (notes/tasks/projects/objectives/habits/inbox) con cambio idéntico — drop local interface definition, `import type { XRow } from '@/types/repoRows'`, `import { saveXQueue } from '@/lib/saveQueue'`, `queue: saveXQueue` en el config del factory. `notesRepo.purgeAll:259` cambió `repo.remove(id)` → `repo.removeRaw(id)` para bypass del LRU cap. Diff total ~14 LOC en 6 archivos. Commit único justificable: cambio mecánico uniforme, bisect granular = ruido. Tests existentes verdes sin cambio gracias a `mockResolvedValue` en `beforeEach` + microtask order del executor (`void executeEntry(id)` corre síncrono al primer attempt en mismo tick que `mockResolvedValue(undefined)`).

- **F29.4 — Bypasses notes (`acceptSuggestion` + `dismissSuggestion`):** ambos wrappers usan `updateDoc` con `arrayUnion` (no pasan por el factory). Rewrite con composite keys: `${noteId}:accept-${suggestionId}` y `${noteId}:dismiss-${suggestionId}` para no chocar con `updateMeta` paralelo del mismo `noteId`. `acceptSuggestion` mantiene `setPartialRow` sync para `noteType` (mutación TinyBase optimista) + enqueue con executor que recheca uid + llama `updateDoc(arrayUnion(suggestionId))`. `dismissSuggestion` no muta TinyBase (no había mutación pre-F29 tampoco). Cast `as unknown as Partial<NoteRow>` documentado: `arrayUnion` retorna FieldValue opaque al queue tipado. `notesRepo.test.ts` +`updateDocMock.mockResolvedValue(undefined)` + `saveNotesMetaQueue.clear()` en `beforeEach` para test isolation entre suites.

- **F29.5 — `<PendingSyncIndicator />` + `usePendingSyncCount`:** componente nuevo `src/components/layout/PendingSyncIndicator.tsx` + hook nuevo `src/hooks/usePendingSyncCount.ts`. El hook usa version-counter pattern: module-level `globalVersion` + `bumpVersion` suscrito a los 7 queues al import (subscribers app-lifetime, sin leak por singletons). `subscribe(cb)` fan-out a `allQueues.map((q) => q.subscribe(cb))` returns multi-unsub. `getSnapshot()` retorna `cachedSummary` si `cachedVersion === globalVersion`, sino re-computa iterando los 7 queues, sumando entries no-synced, marcando `hasError` si ≥1 status `'error'`. `ENTITY_LABELS` array indexed por orden de `allQueues` con singular/plural en español: edición de nota / ediciones de nota, nota / notas, tarea / tareas, etc. Returns `{ total, errorCount, byEntity: [{entity, count, hasError}], hasAny }`. El componente usa `@base-ui/react/popover` (Root/Trigger/Portal/Positioner/Popup) con `data-starting-style/data-ending-style` para animaciones. Trigger: `inline-flex h-11 min-w-11` (44×44 a11y tap target). Color amber si `errorCount === 0`, destructive si `> 0`. Label: `${total} pendiente(s)` o `${total} sin guardar`. Popover content: per-entity list (`{count} {entity}`) + badge `ERROR` si `hasError` + 2 botones: `[Reintentar]` (disabled si `errorCount === 0`, llama `retryNow` por entry en error de cada queue) + `[Descartar]` (`q.clear()` sobre los 7). Iconos `CloudOff`/`RefreshCw`/`Trash2` de lucide. Mounted en `Sidebar.tsx` antes del footer-div con guard `{!collapsed && <div className="px-2 pb-1"><PendingSyncIndicator /></div>}` y en `MobileHeader.tsx` después del `<h1 flex-1 truncate>`.

- **F29.6 — Flush hooks extendidos a los 7 queues (`useSaveQueueFlush.ts`, `useCloseToTray.ts`):** `useSaveQueueFlush` itera `allQueues` con `reduce` para `totalPending()` y `Promise.allSettled(allQueues.map((q) => q.flushAll()))` para `flushAll()`. Listeners `beforeunload` + `online` con guard `totalPending() === 0`. `useCloseToTray` race interno cambia a `Promise.allSettled([...allQueues.map(q => q.flushAll())])` vs `FLUSH_TIMEOUT_MS=2000`. El timeout cubre el agregado, no per-queue.

E2E validado en producción local con Playwright MCP en sesión de cierre F29.7:

- **Indicator amber con mixed pending**: forzado vía `import('/src/lib/saveQueue.ts?t=hash')` desde la consola con executors que nunca resuelven, sobre 4 entries en 3 queues (1 nota + 2 tareas + 1 hábito) → "4 pendientes" en amber, popover lista per-entidad con plural correcto + Reintentar disabled.
- **Indicator destructive**: enqueue con executor que rechaza con `Error` plano transient → tras ~7s (1+2+4 backoff) entry alcanza status `'error'` con attempts=4 → indicator `bg-destructive/15` con "1 sin guardar", popover muestra badge ERROR rojo + Reintentar habilitado.
- **Descartar**: limpia los 7 queues, indicator desaparece del DOM.
- **Backoff con recovery**: enqueue con executor que falla 1 vez y luego succeed → entry transitiona a synced y se limpia del queue tras 1.5s.
- **Toggle favorito real online**: setDoc resuelve, queue queda vacía, indicator no aparece (esperado), UI reordena nota por updatedAt.

Sign-out mid-retry guard, `updateTask({ noteIds })`, `markProcessed`, y `purgeAll` con `removeRaw` cubiertos lógicamente por unit tests (4 nuevos en `baseRepo.test.ts` + el wiring común vía factory).

## Lecciones

- **Vite dev sirve módulos con cache key (`?t=hash`) — dynamic imports cross-extension dan instancias diferentes.** El alias `@/lib/saveQueue` resuelve a `/src/lib/saveQueue.ts?t=hash` (el hash bumpeado por HMR). Un `import('/src/lib/saveQueue')` (sin extensión) o `import('/src/lib/saveQueue.ts')` (con extensión, sin query) resuelven a URLs distintas → instancias DIFERENTES del singleton. Validado en QA E2E F29.7: inyectar entries vía `import('/src/lib/saveQueue.ts')` NO actualiza el indicator del app porque el hook tiene su propia instancia. Solución: leer `performance.getEntriesByType('resource')` para encontrar la URL exacta con `?t=hash` que el app cargó, y hacer `import` con esa URL. Aplicable a futuros tests de cualquier singleton inyectado vía dev console.

- **Composite keys en queues genéricos resuelven el "multi-write per entidad sin choque".** Cuando el wrapper opera sobre un sub-recurso (suggestion dentro de una nota, día dentro de un hábito), encolar con la id raw choca: si el usuario hace `updateMeta(noteId)` y `acceptSuggestion(noteId, sId)` en la misma sesión offline, ambos comparten key `noteId` → upsert reemplaza el primero. Composite `${noteId}:accept-${sId}` separa el namespace sin agregar API surface. Patrón aplicable a cualquier futuro queue donde la id de entidad NO es suficiente para deduplicar el work item.

- **Test isolation entre suites requiere `clear()` separado de `dispose()`.** Module-level singletons (como los 7 queues) viven entre tests del mismo file. Sin reset, una suite que enqueue contamina la próxima. `dispose()` no sirve porque silencia futuros enqueues — la próxima suite no puede usar el queue. `clear()` resetea contenido + notifica subscribers + mantiene `disposed=false`. Patrón aplicable a cualquier singleton stateful expuesto al test runner.

- **`subscribe` app-lifetime + `getSnapshot` cached-by-version es el patrón canónico para `useSyncExternalStore` agregado.** Cuando el hook agrega múltiples sources, `subscribe(cb)` debe fan-out a todos los sources con multi-unsub, y `getSnapshot()` debe ser cacheable: si nada cambió desde el último call, devolver la misma referencia (Object.is requirement de React). El truco: `bumpVersion` suscrito a los queues al module load incrementa un counter cada notify; `getSnapshot()` re-computa solo si `cachedVersion !== globalVersion`. Match exacto al patrón de `useOnlineStatus` pero extendido a N sources. Aplicable a futuros agregadores de stores (ej: tener `usePendingTaskCount` que combine queues + tasks store + analytics).

- **Sign-out mid-retry guard es 5 LOC con valor cross-user defense-in-depth.** El backoff exponencial puede correr durante 1-2 minutos en el peor caso (4s del último attempt + posible reintentar manual). Si el usuario se desloggea, el `auth.currentUser` cambia pero el closure del executor ya capturó el `uid` viejo. Recheck `auth.currentUser?.uid !== capturedUid` al inicio de cada attempt cubre el caso sin agregar API. Throw genérico (NO FirebaseError → no permanent → 4 attempts → 'error') hace el problema visible al usuario via indicator destructive. Aplicable a cualquier retry async donde el contexto de auth puede cambiar entre attempts.

- **`removeRaw` método separado de `remove` resuelve el conflicto bulk-vs-queue sin disable global.** Cuando un repo tiene queue para writes individuales pero usa bulk `Promise.allSettled` para purge/migration, encolar todo el bulk hits LRU cap del queue (50 max). Disable per-call vía flag agrega ruido (`update(id, partial, { skipQueue: true })`); método dedicado `removeRaw` mantiene la API obvia: "si querés retry, usá `remove`; si necesitás bypass, usá `removeRaw`". Aplicable a cualquier futuro repo con bulk operations conviviendo con writes individuales.

- **Single commit vs N commits para cambios mecánicos uniformes.** El plan inicial era 6 commits (uno por repo) para bisect-friendly trail. Refinamiento en Plan mode: si los 6 cambios son IDÉNTICOS (mismo diff modulo nombre de la queue), bisect a nivel de commit no da info útil — el bisect-friendly ya viene del scope de F29.3 ("wiring de queue"). Single commit `feat(repos): wire retry queue en notes/tasks/projects/objectives/habits/inbox` es más legible. Regla: si el commit message describe el patrón uniforme y el diff es realmente uniforme, bundlear. Si hay divergencias per-repo (one tiene un caso edge), separar.

- **QA E2E sintético (inyección de entries pendientes) cubre los escenarios visuales sin offline simulation.** Para validar visual del indicator + popover + Reintentar/Descartar, NO necesito Network throttling real ni intercept de XHR. Inyectar entries con executors que nunca resuelven (`() => new Promise(() => {})`) o que rechazan transient (`() => Promise.reject(new Error('...'))`) cubre los estados pending/retrying/error. Combinado con un toggle real online al final (verificar que happy path no rompe), valida los flows críticos en ~5 minutos vs offline real que requiere DevTools manual + esperar timeouts. Aplicable a futuros tests de UI reactiva a queues async.

## Out of scope (deuda explícita, escalada a ESTADO-ACTUAL)

- **Creates con retry queue.** `createNote/Task/Project/Objective/createFromInbox` quedan con silent loss. Razón: rompe contrato `Promise<string>` del navigate-on-create. SPEC futuro si surge demanda — opciones: encolar create + fast-resolve con UUID local, o mostrar toast de error al caller.

- **`projects/[projectId]/page.tsx:144 + :172` raw `updateDoc` calls.** Bypassan `notesRepo` con `updateDoc(doc(db, 'users/.../notes/...'))` directo. Detectado en Plan mode audit. Fuera del scope F29 porque no usan el factory. Deuda explícita: migrar a `notesRepo.updateMeta` para ganar retry transitivamente.

- **Composite key visible plurality.** El indicator cuenta entries por queue, no por entidad raw. `updateMeta(noteId)` + `acceptSuggestion(noteId, sId)` simultáneos = "2 notas pendientes" aunque sea 1 nota con 2 writes. De-dupe por noteId requiere extra `Set<string>` por entidad — no vale para v1.

- **Persistencia del queue en IndexedDB / Firebase offline persistence.** Mismo argumento que F28: no bloqueante, esperar señales de logs en producción.

- **Telemetría / observabilidad.** Cuántos retries, distribución de error codes, qué entidades fallan más. Útil para dimensionar IndexedDB persistence.

- **Banner per-entidad rico** (toast lib, action sheets, modal de detalle de pendientes). El indicador global de F29.5 cubre la primera versión.

- **Subscription granular per-id en `usePendingSyncCount`.** El `useSyncExternalStore` agregado notifica todas las suscripciones cuando muta cualquier queue. No crítico para v1 (típicamente 0-1 entry pendiente). Refactor a notify selectivo si aparece como bottleneck.

- **`setPartialRow` stale en TinyBase tras `Descartar todo`.** El `clear` de cada queue mata el setDoc/deleteDoc pendiente, pero el `setPartialRow`/`delRow` sync-previo NO se rollbackea. Mismo gotcha que F28 expuso para `saveContent`. Solución completa requiere snapshot pre-write + restore — deuda compartida con F28.

- **Bulk operations (`purgeAll`) sin queue.** Si bulk delete falla un chunk, retry manual por el usuario. Si surge frustración, extender.
