# SPEC — Feature 61: Purga del HTTP cache del WebView en update de APK (Android) (Registro de implementación)

## Objetivo

Tras instalar una APK nueva de release, la app Android seguía mostrando el **bundle viejo** hasta que el usuario cerraba la app y borraba **"Caché"** (no "Datos") en Ajustes de Android. El objetivo fue eliminar esa intervención manual: que un update de APK propague el bundle nuevo en el primer arranque, sin tocar los datos del usuario.

**Causa raíz (verificada en el fuente de Capacitor 8.3.0):** `WebViewLocalServer.java` sirve los assets locales (`index.html` + chunks) con `Cache-Control: no-cache` —que **permite almacenar**, solo pide revalidar— y nunca llama `setCacheMode` (queda `LOAD_DEFAULT`), sin emitir validadores `ETag`/`304` efectivos. El WebView guarda `index.html` (nombre **estable** entre releases) en su HTTP disk cache y lo reusa tras el update → carga los chunks JS hasheados **viejos** → bundle viejo entero. Vive en la partición **"Caché"** (por eso "Eliminar caché" lo curaba sin tocar "Datos": IndexedDB/localStorage/Firestore). No es el Service Worker (no se registra en Capacitor). Upstream sin fix (ionic-team/capacitor discussion #7790, abierta; igual en Capacitor v5/v6/v7). El approach replicó en Android el patrón "purge-on-version-bump" de Tauri (`version_check.rs`): la primera APK con este fix ya autocura a los usuarios trabados (marca ausente → 0 < versionCode actual → purga en el primer arranque).

## Qué se implementó

- **F1 — Gate de versión + purga del HTTP cache:** se agregó un override de `onCreate` en `MainActivity.java`. Gate `BuildConfig.VERSION_CODE > stored` (SharedPreferences propio `secondmind_native`, key `lastPurgedVersionCode`, default 0). Si sube: `getBridge().getWebView().clearCache(true)` + `webView.post(webView::reload)`, y persiste el versionCode con `.commit()` (síncrono). Guard `getBridge()`/`getWebView()` null → difiere sin marcar. Requirió habilitar `buildFeatures { buildConfig = true }` en `android/app/build.gradle` (ver D6). Archivos tocados: `android/app/src/main/java/com/secondmind/app/MainActivity.java`, `android/app/build.gradle`.
- **F2 — Telemetría logcat:** trazas con tag `SecondMindCachePurge` (`gate:` con current/stored/debug, `noop:`, `purged:`, `bridge null`), mismo edit que F1. Archivos tocados: `android/app/src/main/java/com/secondmind/app/MainActivity.java`.
- **F3 — QA on-device + cierre SDD:** se corrieron los Casos A–D on-device (bundle nuevo sin borrado manual + sesión Firebase/IndexedDB intactos), se corrigió el párrafo Android de `Spec/gotchas/pwa-offline.md`, se escaló el gotcha nuevo y se actualizó `Spec/ESTADO-ACTUAL.md`. Archivos tocados: `Spec/gotchas/pwa-offline.md`, `Spec/ESTADO-ACTUAL.md`, este SPEC.

## Decisiones clave

| ID     | Decisión                                                                            | Rationale                                                                                                                                                                                                                                                                                                                                                     |
| ------ | ----------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **D1** | Override de `onCreate`, no de `load()`                                              | Tras `super.onCreate()`, `getBridge()` ya es no-null y la navegación inicial (`loadUrl(appUrl)` síncrono en `BridgeActivity.load()`) ya se disparó. `load()` es `protected` y overridearlo obliga a replicar su contrato — más frágil sin ganancia.                                                                                                           |
| **D2** | Reload posteado `webView.post(webView::reload)`                                     | Se serializa **después** del `loadUrl` inicial síncrono, ya con el cache purgado. `WebView.reload()` re-valida la URL actual (no re-navega a la raíz como `Bridge.reload()`, que hace `post(loadUrl(appUrl))`). No re-dispara `onCreate` → sin loop infinito. Doble-load brevemente visible solo post-update; el splash (F40 `launchAutoHide:false`) lo tapa. |
| **D3** | Gate `>` (monotónico), no `!=`                                                      | CI computa `versionCode = MAJOR*10000+MINOR*100+PATCH` (estrictamente creciente). `>` no purga en downgrade/rollback (ese cache ya se invalidó al subir). Primera instalación: stored ausente → 0 → purga inocua sobre cache vacío.                                                                                                                           |
| **D4** | `.commit()` (síncrono), no `.apply()`                                               | Corre en main thread en `onCreate`; un int es trivial y elimina la ventana de carrera (proceso muere antes del flush async de `apply`). Marca **solo al purgar** (dentro del `if`) → un fallo de purga no marca la versión como ya purgada.                                                                                                                   |
| **D5** | SharedPreferences propio `secondmind_native`                                        | No pisar la key de Capacitor `LAST_BINARY_VERSION_NAME` (que Capacitor usa en `Bridge.isNewBinary()` para invalidar `CAP_SERVER_PATH` de live-updates, NO el HTTP cache). La purga es complementaria, no redundante.                                                                                                                                          |
| **D6** | Habilitar `buildFeatures { buildConfig = true }` en `build.gradle` (desvío al plan) | El plan asumió que `BuildConfig` se generaba siempre (AGP 8.13). **Falso:** desde **AGP 8.0 la generación de `BuildConfig` está OFF por default**. Sin el flag, `MainActivity` no compila (`cannot find symbol: variable BuildConfig`). Atrapado al compilar de verdad (regla "probar el verificador"). Cambio mínimo y estándar; no afecta otras variantes.  |

## Lecciones

- **El HTTP disk cache del WebView Android sirve el bundle viejo tras un update de APK** porque `index.html` tiene nombre estable entre releases y Capacitor lo sirve con `Cache-Control: no-cache` (permite almacenar, solo revalidar) sin validadores `ETag`/`304` efectivos. Vive en la partición "Caché" (no "Datos"), por eso "Eliminar caché" lo curaba sin tocar IndexedDB/localStorage/Firestore. La cura es un gate `BuildConfig.VERSION_CODE > stored` en `onCreate` (SharedPreferences fuera del WebView) → `clearCache(true)` + `reload`. Detalle canónico escalado a `Spec/gotchas/pwa-offline.md`.
- **`BuildConfig` no se genera por default desde AGP 8.0** — requiere `buildFeatures { buildConfig = true }` explícito en `build.gradle`. El plan asumió generación automática y `MainActivity` no compiló (`cannot find symbol: variable BuildConfig`); lo atrapó compilar de verdad en vez de confiar en la asunción. Generaliza: validar las asunciones de build-tooling compilando, no leyendo docs de una versión anterior.
- **El patrón "purge-on-version-bump" generaliza cross-platform** — la misma estrategia de Tauri (`version_check.rs`, F36) se replicó en Android con un gate de versión persistido fuera del WebView. La primera build con el fix autocura a los usuarios ya trabados (marca ausente → default 0 < versionCode actual → purga en el primer arranque), sin necesidad de un release intermedio.
- **`WebView.reload()` vs `Bridge.reload()`:** el primero re-valida la URL actual sin re-navegar a la raíz ni re-disparar `onCreate` (evita loop infinito); el segundo hace `post(loadUrl(appUrl))`. Para purgas post-cache la diferencia es load-bearing.
