## Objetivo

Cuando el usuario abre cualquier popup del editor (slash menu o wikilink menu) y luego scrollea el documento, el popup se cierra de inmediato — alineado con el comportamiento de Notion/Linear/Slack. Adicionalmente, ambos popups comparten una misma base de positioning vía `@floating-ui/dom` con `flip` y `shift`, eliminando la divergencia previa donde el wikilink popup quedaba desalineado tras scroll y el slash popup perseguía al user. Beneficio meta: cualquier popup futuro del editor (autocompletado de tags, bloques de código sugeridos, etc.) hereda el comportamiento correcto al consumir el mismo hook.

## Decisiones clave

| Decisión                                                  | Opciones                                 | Elegida          | Razón                                                                                                                                                                                                     |
| --------------------------------------------------------- | ---------------------------------------- | ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Hook genérico `<TItem>`                                   | uno por popup                            | Genérico         | El shape común es el lifecycle, no los datos. Hooks separados duplicarían todo el lifecycle y derrotarían el propósito del refactor.                                                                      |
| Lifecycle 100% encapsulado                                | exponer cleanup hooks al consumer        | Encapsulado      | Pedido explícito del user. Si el consumer puede olvidarse de algo, eventualmente alguien va a leakear.                                                                                                    |
| `autoUpdate` con `ancestorScroll: false`                  | autoUpdate completo + onClose en scroll  | Separado         | autoUpdate con scroll true es lo que causaba la persecución. Desactivar `ancestorScroll` corta la persecución; un handler `scroll` en `document` agrega el cierre. Dos responsabilidades, dos mecanismos. |
| Close on scroll vs close on cursor off-viewport           | detectar bounding rect cursor cada tick  | Close on scroll  | Más simple y predecible. Detectar cursor off-viewport requiere computar `getBoundingClientRect` cada tick y es subjetivo (¿qué es "off"?).                                                                |
| Scroll listener en `document` capture vs `window` capture | window capture                           | document capture | Semánticamente explícito. Cualquier `<main>` interno con overflow scrollable propio es atrapado durante la fase capture en `document`.                                                                    |
| Filter `queryItems` síncrono                              | async para casos hipotéticos             | Síncrono         | Todos los items actuales (slash hardcoded, wikilink desde TinyBase) son in-memory. Si emerge un caso async (ej. embeddings), se extiende entonces.                                                        |
| `PopupListener<TItem>` único                              | preservar tipos específicos como aliases | Único            | Aliases serían deuda visual sin valor (mismo shape, distinto nombre). Eliminarlos completa la unificación a nivel tipo.                                                                                   |
| `@testing-library/react` + jsdom                          | montar componente dummy                  | RTL + renderHook | Patrón estándar de la industria, tests legibles, dep que se usará para futuros tests de hooks del proyecto.                                                                                               |

## Rondas de fix

**Bug detectado en E2E F5: el slash menu abría y se cerraba en el mismo tick.** Causa raíz: ProseMirror invoca `scrollIntoView()` sincrónicamente al insertar el char trigger (`/` o `@`) para mantener el cursor visible. El listener de close-on-scroll, registrado inmediatamente al `onStart`, atrapaba ese scroll programático y cerraba el popup antes de que el user lo viera. Síntoma observado: `mainScrollTop: 582` apenas se presionaba `/`, y el popup nunca era visible.

**Fix:** registrar el listener con `setTimeout(0)` — defer al próximo task tick. Para entonces el `scrollIntoView` ya terminó (es sincrónico durante el handler del press). Los scrolls subsiguientes del user (mouse wheel, trackpad, touch) sí caen en el listener registrado y cierran el popup como esperado. Tests L4 ajustados con `act(async)` + `await new Promise(setTimeout, 0)` para flush del defer antes de verificar la registración del listener vía spy. 24/24 tests siguen verdes. Commit `73392cb`.

## Lecciones

- **Eventos `scroll` no burbujean por design del DOM.** Listener en `document.addEventListener('scroll', ..., { capture: true })` SÍ intercepta scrolls de cualquier descendant durante la fase capture — incluido un `<main>` interno con overflow propio. Verificado en este repo: el shell tiene `<main class="flex-1 overflow-auto">` que es el scrollable real (`<body>` no scrollea), y el listener atrapó ambos: scroll desktop con mouse wheel y touch scroll mobile en 375px. **Regla:** nunca registrar `scroll` listener sin `capture: true` cuando se busca capturar scrolls del documento entero. El fallback `getOverflowAncestors` de `@floating-ui/dom` (planeado en el SPEC original como Plan B) no fue necesario.
- **ProseMirror invoca `scrollIntoView` sincrónicamente al insertar el char trigger de un Suggestion.** Cualquier handler de scroll registrado en el mismo tick que el `onStart` del listener captura ese scroll programático y dispara cierre prematuro del popup. **Regla:** popups que abren tras un char trigger (slash, mention, autocomplete) deben deferir la registración del scroll handler al menos un task tick (`setTimeout(0)`). Aplicable a cualquier futuro popup similar — el comportamiento de ProseMirror es genérico, no específico de slash o wikilink.
- **El listener pattern module-level (`let activeListener = null` + `setXMenuListener`) funciona porque TipTap Suggestion plugin tiene un solo instance activo a la vez.** Si en el futuro se necesitan dos instances simultáneas del mismo popup (ej. nested editors), el módulo-level singleton se rompe. Mitigación preventiva: el hook ya capta esta limitación en el comentario del file. Si emerge, refactor a un `Map<PluginKey, Listener>` keyed por `pluginKey`.
- **Convención del repo para hooks:** `interface UseXReturn { ... }` exportada + return type explícito en función + named object (no tuples). Hooks en este proyecto: `useNoteSearch`, `useStoreInit`, `useStoreHydration`, `useAutoUpdate`, ahora `useEditorPopup`. Naming `useXY` con explícito return type ayuda a TS strict y a la legibilidad. Los nuevos hooks deben seguirla.
