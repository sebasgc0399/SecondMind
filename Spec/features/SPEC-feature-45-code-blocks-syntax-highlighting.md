## Objetivo

Hasta F45 los code blocks del editor TipTap eran `<pre><code>` monoespaciado plano sin distinción visual de tokens ni metadata de lenguaje — ilegibles apenas el snippet pasaba 5 líneas. F45 los convierte en bloques enriquecidos estilo Notion/Linear: pill de lenguaje arriba-izquierda con dropdown (Auto + 25 lenguajes), syntax highlighting de tokens con paleta IDE-like (rosa-violeta keywords, verde strings, cyan tipos, naranja números, terra-cotta atributos, gris itálico comments), botón copy arriba-derecha con feedback inline.

## Lecciones

- **highlight.js peerDep mismatch silencioso post-npm install.** Al agregar `lowlight@^3` + `@tiptap/extension-code-block-lowlight`, npm respeta el `highlight.js@10.7.3` ya en el lockfile en vez de bumpear a `^11` (que ambas deps piden como peer). `npm ls highlight.js` muestra `invalid` warning pero el install es exit 0. Síntoma runtime: `lowlight.highlightAuto()` lanza `TypeError: Cannot read properties of undefined (reading 'variants')` en `expandOrCloneMode` porque la API de modes cambió entre v10 y v11. Fix: bump explícito `npm i highlight.js@^11`. Aplica a cualquier dep con peer `highlight.js@^11` introducida en un repo cuyo lockfile arrastra v10 desde otro consumer (firebase-tools u otra trae `marked-terminal` que depende de highlight.js, contaminación cross-package). Lección operativa: post-`npm install` de nueva dep que pide peer, **siempre** correr `npm ls <peer>` y verificar resolved version contra el peerDep declarado — el `invalid` warning de npm es load-bearing, no decorativo.

- **Vite cachea node_modules/.vite/deps aunque cambies versiones de deps.** Tras bumpear `highlight.js` v10→v11, el dev server siguió sirviendo el bundle viejo de `lowlight.js?v=80a6f965` (mismo hash query) — el error de `variants` persistió hasta hacer `rm -rf node_modules/.vite && kill dev server && npm run dev`. Vite no detecta el cambio porque el dep path no cambió, solo el contenido transitivo. Pattern operativo: tras cualquier cambio de dep transitiva relevante (peerDep upgrade, sub-dep override), borrar `node_modules/.vite/` antes de reiniciar el dev server. Equivalente del problema de `resolve.dedupe` reincidente tras npm install (CLAUDE.md), pero con la cache de Vite en lugar del lockfile.

- **`tauri-plugin-single-instance` no es el único que sabotea ciclos de dev.** El kill del dev server vía TaskStop puede dejar el proceso `node` huérfano (port 5173 ocupado). En Windows: `Get-NetTCPConnection -LocalPort 5173 | Stop-Process -Force -Id $_.OwningProcess`. Aplica a cualquier background server (Vite, Express, etc.) cuyo wrapper bash muere antes de que el child node reciba SIGTERM. Documentar en operativos sólo si reincidente — primera vez es ad-hoc.

- **Primer NodeView React del proyecto (`nodeviews/CodeBlockNodeView.tsx`) sienta el patrón:** `<NodeViewWrapper className="...">` con header `contentEditable={false}` (controles que no compiten con la selección del editor) + `<NodeViewContent<'code'> as="code" className="font-mono" />` para el slot editable. El generic `<'code'>` es **obligatorio en TypeScript estricto** — sin él TS infiere default `'div'` y rechaza `as="code"` con `TS2322`. `ReactNodeViewRenderer(CodeBlockNodeView)` se aplica vía `.extend({ addNodeView() { ... } })` antes de `.configure(...)`. Las clases `.hljs-*` que emite lowlight se aplican dentro del `<code>` automáticamente — no requieren intervención del NodeView.

- **`@base-ui/react/select` value `null` es legal — handler debe aceptarlo.** `Select.Root onValueChange` type es `(value: string | null, eventDetails: SelectRootChangeEventDetails) => void` (no solo `string`). Aún si el componente nunca emite `null` en la práctica (siempre que items tengan `value`), TS exige el handler con la signature completa. Patrón: `const handle = (next: string | null) => { if (next === null) return; ... }`. Aplica a cualquier futuro `Select` del proyecto.

- **El swap visual del botón copy puede no observarse vía Playwright MCP (`navigator.clipboard.writeText` retorna pero el `.then` puede correr fuera del timing del check siguiente).** El clipboard real sí recibe el texto — verificado leyendo `navigator.clipboard.readText()` post-click. Lo que no se observó: el `setCopyState('copied')` causando re-render del icon a `Check`. En browser nativo el feedback se ve OK (verificable manualmente). Si la regresión emerge en QA real, investigar si `node` (prop TipTap) cambia ref tras `updateAttributes()` (re-render del NodeView con instancia nueva) y romper el closure del `useCallback` cleanup. Mitigación posible: `const nodeRef = useRef(node); useEffect(() => { nodeRef.current = node; }, [node]);` y leer `nodeRef.current.textContent` en lugar de capturar `node` en deps. **No bloqueante** para F45 — feature funcional, copy real funciona.
