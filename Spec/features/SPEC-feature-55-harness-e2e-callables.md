## Objetivo

Construir el primer harness E2E de callables del repo: una suite (`npm run test:functions`) que levanta los emuladores de functions + auth + firestore, invoca las 5 callables de access reales vía client SDK con tokens de verdad (`httpsCallable` + `connectFunctionsEmulator` + `connectAuthEmulator`), y afirma su comportamiento de punta a punta — cliente → gate de auth real (`request.auth`) → I/O Firestore → respuesta. Cubre lo que el unit no puede: que el `requireAdmin`, el capacity transaccional, la idempotencia, el no-oráculo y el rate-limit se comporten con un token real atravesando el runtime de la CF.

Causa-raíz que cerró: el "E2E emulador 18/18" del registro de SPEC-52 era smoke manual, no código commiteado — no existía suite automatizada que invocara las callables reales contra el emulador. El disparador del draft era construirlo antes de abrir la beta a escala (`0.5.0`) o si entraba un 2º dev; este SPEC deja el harness listo para activar cuando suba el costo de un error. Alcance = batch 1 (las 5 callables de access); BYOK (`saveApiKey`/`deleteApiKey`) quedó para batch 2 (requiere migrar a imports modulares).

- **F4 — Inyección de `ADMIN_EMAIL` vía `.secret.local` generado en runtime (D5):** `scripts/emu-secret.mjs` escribe `src/functions/.secret.local` con `ADMIN_EMAIL=admin-e2e@secondmind.test` antes de `emulators:exec`, porque `defineSecret().value()` resuelve de Secret Manager o del `.secret.local`, no de `process.env`. Gitignoreado (`*.local` raíz) → se regenera cada corrida, cero secret commiteado. Verificado por un assert E2E (`listAllowlistMembers` como admin devuelve `{members}`, no `permission-denied`). - **F5 — Bootstrap cliente↔emulador + usuarios de prueba:** helpers `e2e/helpers/emulator.ts` (app cliente apuntando a los emuladores, región `us-central1`, `projectId: 'demo-secondmind'`) y `e2e/helpers/users.ts` (`createTestUser` + login que devuelve una `Functions` autenticada con token real; perfiles admin/no-admin/sin-sesión). `emailVerified` NO se setea (ninguna de las 5 callables lee `token.email_verified`; ese gate vive en las security rules, que el harness no atraviesa). Archivos tocados: `e2e/helpers/emulator.ts` (nuevo), `e2e/helpers/users.ts` (nuevo).
- **F6 — Seed + cleanup de Firestore entre casos:** `e2e/helpers/firestore.ts` reusa `@firebase/rules-unit-testing` (D7): `beforeEach(clearFirestore)` + seed vía `withSecurityRulesDisabled` para `accessRequests/`, `allowlist/` y `config/app`; mismo `projectId` que el runtime de functions (singleProjectMode) para compartir Firestore. Archivos tocados: `e2e/helpers/firestore.ts` (nuevo).
- **F7 — `checkMyAccess` (canario):** casos allowlisted→`{authorized:true}`, no-allowlisted→`false`, sin sesión→`unauthenticated`, y no-oráculo (un email de tercero en `data` se ignora, responde sobre el propio token). Archivos tocados: `e2e/checkMyAccess.e2e.test.ts` (nuevo).
- **F8 — `listAllowlistMembers` (gating admin + DTO):** admin→`{members:[{email,addedAt}]}` ordenado por `addedAt` desc; no-admin→`permission-denied`; sin sesión→`unauthenticated`. Archivos tocados: `e2e/listAllowlistMembers.e2e.test.ts` (nuevo).
- **F9 — `revokeAccess` (borrado + idempotencia + gating):** admin borra `allowlist/{email}` (verificado leyendo Firestore post-llamada); idempotente sobre email inexistente; gating no-admin/sin-sesión; `invalid-argument` para email vacío/no-string. Archivos tocados: `e2e/revokeAccess.e2e.test.ts` (nuevo).
- **F10 — `processAccessRequest` (capacity transaccional, el crítico):** aprobación bajo límite, `resource-exhausted` sobre límite con tx abortada (allowlist no crece, request sigue `pending`), idempotencia con beta llena + email ya en allowlist (no consume slot), reject, `not-found`, gating y `invalid-argument`. - **F11 — `submitAccessRequest` (pública, no-oráculo, dedup, rate-limit):** pública sin sesión→`{ok:true}` crea `accessRequests/{email}`; no-oráculo (respuesta idéntica para email nuevo/duplicado/ya-allowlisted); dedup (no pisa `createdAt`/`status`); rate-limit por IP (todas las llamadas del cliente caen al mismo bucket en el emulador → superar `perMinute` da `resource-exhausted`); `invalid-argument` por email/motivo malformado. Archivos tocados: `e2e/submitAccessRequest.e2e.test.ts` (nuevo).

## Lecciones

- **`defineSecret().value()` no resuelve de `process.env` en el emulador** — resuelve de Secret Manager (prod) o de `<functions>/.secret.local` (emulador). Inyectar secrets al runtime del emulador requiere generar ese archivo en runtime (gitignoreado, regenerado por script), no un env-var inline ni `cross-env`: la var quedaría en el proceso del script, no en el runtime forkeado del emulador. Generalizable a cualquier futuro harness de CF sobre emulador.
- **El emulador de functions sirve `lib/index.js`, no el `src/`, y `emulators:exec` NO corre el `predeploy` de `firebase.json`** — hay que compilar functions explícitamente antes de levantar el emulador, o la suite corre contra un build viejo/inexistente. El script `test:functions` lo encadena.
- **Para ejercer el gate de auth real (token → `request.auth`) el client SDK con Auth emulator es la única vía** — `firebase-functions-test` mockea el `CallableRequest` y nunca atraviesa ese flujo, dejando abierto justo el hueco que el unit de `requireAdmin` no cubre. El costo es boot del emulador (lento) → CI, no pre-commit.
- **El harness de callables NO atraviesa las security rules** (el Admin SDK las bypassa) → los invariantes que viven en rules (p.ej. `email_verified`) no se cubren acá; este harness valida solo el gate aplicativo de la CF (`request.auth`, `requireAdmin`, I/O Firestore). Saber qué capa cubre cada suite evita falsos verdes.
- **El rate-limit por IP del emulador colapsa todas las llamadas del cliente al mismo bucket** → el límite se gatilla con submits consecutivos y es afirmable E2E. Regla operativa: validar empíricamente el comportamiento de IP del emulador antes de escribir el caso; si no expone una IP diferenciable, documentar el límite de cobertura en vez de inventar el caso.
- **Hallazgo de CI fuera de scope, arreglado de paso:** el CI venía rojo desde SPEC-52/53 porque el `npm ci` del root no instalaba las deps de `src/functions` (los unit de functions fallaban solo en CI con `ERR_MODULE_NOT_FOUND`). Lección general: un monorepo con sub-package de functions necesita instalar/buildear ese sub-package explícitamente en CI; el `npm ci` del root no lo cubre. Gotchas operativos vigentes en `Spec/gotchas/cloud-functions-guards.md` y `Spec/gotchas/tooling-local.md`.
