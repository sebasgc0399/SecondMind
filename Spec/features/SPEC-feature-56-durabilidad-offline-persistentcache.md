## Objetivo

Hacer que SecondMind sobreviva offline a un reload **sin pérdida de datos ni de visibilidad**: tras un reload/relaunch sin red, (a) los datos del usuario **siguen visibles** (no app vacía) y (b) los writes hechos offline **se replayan y persisten** al reconectar. Se logró delegando durabilidad + replay + lectura offline al **SDK de Firestore** vía `persistentLocalCache` (IndexedDB), reemplazando el `getFirestore(app)` plano. Cambio mínimo (~10–15 LOC), reversible en un commit, sin tocar el patrón de datos del repo.

**Causa-raíz original:** `getFirestore(app)` plano (cache en memoria) + TinyBase sin persister local + `getDocs()` offline vacío → doble pérdida offline tras reload: writes pendientes (saveQueues en memoria) + app sin datos. Peor que vacía: el `.catch` de `useStoreInit` solo loggeaba → `isHydrating` quedaba `true` → splash colgado indefinido.

## Qué se implementó

- **F1 — `initializeFirestore` con `persistentLocalCache`:** reemplazó `getFirestore(app)` por `initializeFirestore(app, { localCache: persistentLocalCache({ tabManager: persistentMultipleTabManager(), cacheSizeBytes: 40*1024*1024 }) })`. El SDK asume durabilidad de writes (mutation-queue durable) + lectura offline. Archivos tocados: `src/lib/firebase.ts`.
- **F1.1 — Guard offline en `useStoreInit`:** el `.catch` resuelve la hidratación (`setHydratedUserId`) ante CUALQUIER error de `startAutoLoad`/`getDocs` (invariante "nunca colgar el splash", Opción A), respetando el guard de double-mount (`cancelled` + `currentUserIdRef`). Archivos tocados: `src/hooks/useStoreInit.ts`.
- **F2 — `navigator.storage.persist()` en boot:** helper idempotente best-effort, loggea el retorno (`[storage:persist]`). Archivos tocados: `src/lib/storagePersist.ts` (nuevo), `src/main.tsx`.
- **F3 — Schemes nativos congelados (verify-first):** `useHttpsScheme: false` explícito en ambas windows Tauri (origin `http://tauri.localhost`, verificado en el build de prod antes de fijar); Capacitor `androidScheme: 'https'` confirmado (origin `https://localhost`). Invariante anti data-loss. Archivos tocados: `src-tauri/tauri.conf.json`, `capacitor.config.ts`.
- **F4 — Documentación de invariantes + kill-switch:** scheme invariante (ambos webviews), read-through (D7), Android best-effort (D8), kill-switch (C1). Archivos tocados: `CLAUDE.md`, `Spec/gotchas/tinybase-firestore.md`, `Spec/gotchas/pwa-offline.md`, `Spec/ESTADO-ACTUAL.md`.
- **F5 / F6 / F7 — Gates de QA** (web → Tauri → Android): ver "Resultados de QA".

## Lecciones

- **`setDoc` no resuelve su promesa offline** (con o sin cache; Context7): el write se aplica al cache local (latency compensation, `hasPendingWrites=true`) pero la promesa espera el ack del server. → la durabilidad de P1 NO depende de que la promesa resuelva (el SDK ya encoló la mutación durable); pero el copy "Guardando…" (P2) sigue colgado offline (ortogonal, no cae gratis con esta feature).
- **`cacheSizeBytes` va DENTRO de `persistentLocalCache({...})`**, no top-level en `FirestoreSettings` junto a `localCache` → **throw en runtime que TS no atrapa** (ambos campos existen en el type). NUNCA `CACHE_SIZE_UNLIMITED`.
- **El scheme del origin en webviews nativos es invariante permanente:** cambiarlo (Tauri `useHttpsScheme` / Capacitor `androidScheme`) reubica la carpeta de IndexedDB + huérfana el localStorage del origin = data loss masivo (Tauri #11252). Verify-first: leer el origin real del build de prod ANTES de congelar, nunca asumir el default.
- **El guard "nunca colgar el splash" no se ejerce desde afuera** con `persistentLocalCache`: el SDK resuelve `getDocs` vacío por el `.then` en todos los escenarios degradados probados (cache vacía, init-failure). El guard queda como defensa-en-profundidad data-safe para rechazos que el SDK no produce hoy; la corrupción severa (`dd85`) es un assertion, tampoco cae en `.catch`.
- **Android es best-effort estructural:** IndexedDB del WebView es transient (el OS reclama storage bajo presión, eviction LRU all-or-nothing); `persist()` puede devolver `false`. El force-stop reproduce el relaunch feliz, NO la eviction bajo presión real → pasar F7 no garantiza cero pérdida en Android.
- **El replay de Android es lento** (~1 min para un write flusheado, vs instantáneo en web/desktop) — verificar el server con espera, no asumir replay inmediato.
- **D13 (hallazgo de F7):** offline, el `saveQueue` serializa por entidad y NO re-dispara `setDoc` para upserts mientras el primero está en vuelo (el re-execute depende del version-check post-`await`, que offline nunca corre) → solo el primer write de cada nota es durable ante un kill. Distinto de D11 (debounce, antes del primer `setDoc`). Fix → SPEC-57, pre-requisito de beta.
