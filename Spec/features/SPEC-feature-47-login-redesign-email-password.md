## Objetivo

El usuario puede crear cuenta o iniciar sesión con email + contraseña (además de Google), recuperar contraseña vía email reset, ve un banner persistente en la app recordándole verificar su email tras sign-up (no bloqueante), recibe mensajes claros cuando intenta usar Google en una cuenta email/password (o viceversa), y queda bloqueado del registro cuando la beta alcanza el límite configurado en Firestore. La pantalla `/login` quedó visualmente alineada con el design system (Linear-style card + radial glow indigo + tipografía hero) en lugar del minimal previo de un solo botón Google.

## Lecciones

- **Email Enumeration Protection (EEP)** altera el comportamiento de Firebase Auth: `fetchSignInMethodsForEmail` retorna array vacío, errores `wrong-password`/`user-not-found` colapsan en `invalid-credential` genérico. Cualquier feature de auth que asuma poder distinguir entre "user no existe" y "password incorrecto" o identificar provider del email DEBE preguntar primero si EEP está activa. Mensajes UX menos precisos a cambio de seguridad real anti-enumeration.
- **`firebase-functions@7.x` mantiene namespace v1 funcional para Auth triggers** (`functions.region().auth.user().onCreate/onDelete`). NO documentado en context7 ni en docs latest — solo aparece v2 blocking `beforeUserCreated` (requiere Identity Platform pago). Para triggers Auth async post-create sin upgrade de plan, v1 sigue siendo la opción viable. Migrar a v2 blocking solo cuando billing lo justifique.
- **`user.reload()` NO dispara `onAuthStateChanged`.** Si necesitás reflejar cambios de `emailVerified` o `displayName` post-reload sin requerir logout, exportá un `refreshUser()` desde el hook que haga `await auth.currentUser?.reload()` + `setUser(auth.currentUser)` para forzar re-render del consumer. Single source of truth en el hook canónico.
- **Email/password NO necesita branching multi-plataforma** (Capacitor/Tauri/web). El SDK web de Firebase (`signInWithEmailAndPassword`, `createUserWithEmailAndPassword`) funciona idéntico en los 3 frentes porque corre dentro del WebView (sin OAuth externo). El branching `isCapacitor/isTauri` del `useAuth.signIn` original es solo necesario para Google (popup restrictions). Extender hooks con métodos nuevos email/password queda flat.
- **`base-ui/react` no usa `data-state` (Radix) sino `data-active` + `data-starting-style`/`data-ending-style`.** Las clases `animate-in`/`animate-out` de tw-animate-css NO aplican a componentes base-ui — animar transitions via Tailwind selectors con `data-starting-style:scale-95 data-ending-style:opacity-0`. Para wrappers `<div>` NO-base-ui (mode switch tabs↔reset en LoginCard), `useMountedTransition` + `animate-in`/`animate-out` SÍ aplica — son contextos ortogonales.
- **`useMountedTransition` requiere `fill-mode-forwards` en animaciones de exit.** Sin él, el componente saliente revierte mid-vuelo a translateX(0) generando snap visual. Gotcha vivo escalado desde F33/F35; aplica a cualquier feature que use el hook para switch de vistas con anim.
- **Pre-mount de Firebase ↔ TinyBase: TinyBase no está disponible sin user.** Cualquier hook que corra en rutas pre-auth (login, reset password, account recovery) debe leer Firestore directamente vía `getDoc` del SDK web — NO via TinyBase (sus persisters dependen del UID). Documentar el bypass al tope del hook nuevo (ver `src/hooks/useSignupCapacity.ts` para el patrón) para evitar "fix bien intencionado" futuro que migra a TinyBase y rompe la pantalla.
- **Counter mantenido por triggers vs `listUsers()` paginado en cada check:** el primero es O(1) read + O(1) increment, el segundo O(n) y costoso post-1000 users. Trade-off: race en signups simultáneos puede dejar beta excedida por 1 puntualmente (counter eventually consistent). Aceptable para single-user-ish beta; si crece, migrar a v2 blocking `beforeUserCreated` para race-safety estricto.
- **`--warning` y `--foreground-muted` tokens del design system MASTER.md NO existen en `src/index.css`.** El runtime real expone `--muted-foreground` (no `--foreground-muted`) y no tiene `--warning` (usar `amber-500` directo, precedente `TaskCard.tsx`). MASTER.md está desincronizado del runtime — siempre verificar tokens existentes con grep en `src/index.css` antes de usar nuevas clases.
- **Política Firebase Console password (server-side)** rechaza con `auth/weak-password` si falta numérico o length<8 — independiente de la validación client-side. El mapper UX debe reflejar la regla COMPLETA del server ("Mínimo 8 caracteres con al menos un número.") para no confundir al user con feedback parcial. Defense in depth: client valida lo mínimo para feedback inmediato, server enforce policy real.
- **Seed inicial obligatorio para counters mantenidos por triggers** post-deploy. Antes de activar los triggers, ejecutar one-shot que inicialice el counter con el estado actual del sistema. Sino el contador empieza en 0 y los datos pre-feature no se cuentan hasta que se borren y recreen (caso que típicamente no pasa en producción → contador queda permanentemente subestimado).
