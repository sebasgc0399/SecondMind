## Objetivo

Hasta F46 el editor de notas ocupaba todo el viewport — comparar dos notas, escribir A mirando B, o reorganizar ideas entre notas requería abrir tabs paralelos del browser (sin estado compartido del cliente — autosave, preferences, sidebar duplicados → pérdida de UX coherente y consumo de memoria 2x). F46 introduce un layout split-pane horizontal nativo: el usuario abre una segunda nota lado a lado dentro de la misma sesión, con handle resizable persistente, validación anti-conflicto (no permitir la misma nota en ambos panes), y degradación clean en tablet/mobile (single pane). El estado del split vive en URL (`?split=noteId`) — bookmarkeable, refresh-safe, share-friendly.

## Decisiones clave

| ID  | Decisión                                                                                            | Razón                                                                                                                                                                                                                                                                                               |
| --- | --------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| D1  | Estado del split en URL (`?split=noteId`) + persistencia del RATIO en preferences (no de qué notas) | Bookmarkeable, refresh-safe, share-friendly. La URL es contextual a la nota navegada — persistir notas across sesiones genera estados confusos. El ratio sí es preferencia global del user.                                                                                                         |
| D2  | Prohibir la misma nota en ambos panes (filtro UI en picker + `console.warn` en URL manual)          | `useNote` es one-shot (no `onSnapshot`) y `useNoteSave` debouncing por noteId — permitir misma nota requeriría rediseñar a sync reactivo (riesgo de loops save→load + complejidad fuera de scope MVP).                                                                                              |
| D3  | `react-resizable-panels` sobre custom DOM                                                           | Lib mantenida (Vercel/shadcn ecosystem), API React-idiomática, accessibility + keyboard resize built-in, persistencia API nativa. Custom resize agregaría ~80 líneas de pointer tracking sin valor diferencial.                                                                                     |
| D4  | Sin snap zones (33/67, 50/50, 67/33) en MVP — drag libre con default 50/50                          | Snap requiere lógica de threshold + animación post-release. MVP valida si el feature aporta valor real; snap es polish posterior.                                                                                                                                                                   |
| D5  | Único atajo MVP: `Cmd/Ctrl+\\` (toggle split). Cerrar pane individual = botón X en header del pane  | `Cmd+W` colisiona con browser (cierra tab), `Cmd+]/Cmd+[` colisionan con browser history + TipTap indent. Focus entre panes = click directo. Shortcuts de focus se evalúan post-MVP si emerge fricción.                                                                                             |
| D6  | Solo desktop (≥1024px). Tablet/mobile ignoran `?split=`, renderizan single pane                     | Espacio horizontal en tablet (768-1023px) genera panes <500px cada uno — incómodos para edición. Mobile sin chance. Botón split hidden en <1024px.                                                                                                                                                  |
| D7  | NO bumpear `PREFERENCES_SCHEMA_VERSION` al agregar `splitPaneLayout` (campo aditivo opcional)       | Bumpear purga preferences existentes (gotcha F36.F8). `splitPaneLayout: { left, right }` required con default `{ left: 50, right: 50 }` en `parseSplitPaneLayout` evita migración y preserva otras prefs del user.                                                                                  |
| D8  | Modal del picker con `@base-ui/react/dialog` directo (sin wrapper shadcn)                           | Consistente con patrón F45 (`@base-ui/react/select` directo). `src/components/ui/` no tiene Dialog base-ui (solo alert-dialog Radix). Evita crear wrapper temprano sin segundo caller que justifique abstracción.                                                                                   |
| D9  | `Group` montado siempre (no fast-path para single mode)                                             | Un fast-path con/sin Group cambia la position del NoteEditorContainer left en el tree React → re-mount al toggle del split → perdería keystrokes pre-flush del autosave 2s. Overhead del Group con un solo Panel (~5-10 kB gz) es aceptable vs UX hit.                                              |
| D10 | 3 entrypoints visuales de toggle coexisten (TopBar + header NoteEditor + CommandPalette)            | TopBar cubre sidebar-oculta solamente; header del editor cubre sidebar-visible (siempre visible junto a DistillIndicator); CommandPalette para power users. 3 callers ahora del toggle inline (URL manipulation + dispatch event); si emerge un cuarto, extraer a `useSplitToggle` hook compartido. |

## Lecciones

- **`react-resizable-panels` v4 API rompió con v2.** `PanelGroup`/`Panel`/`PanelResizeHandle`/`defaultSize`/`autoSaveId` (v2) fueron reemplazados por `Group`/`Panel`/`Separator` + hook `useDefaultLayout({ id, panelIds, storage })` con `LayoutStorage` adapter custom + `defaultLayout: { [id]: flex-grow }` + `onLayoutChanged` per-Group. La opción `panelIds` resuelve nativamente conditional Panel rendering — eliminando la necesidad de key-based remount del PanelGroup completo que requería v2. Cuando context7 sugiere API distinta a la documentación que conocés, validar con `npm view <pkg> version` primero antes de asumir que v2 sigue mainstream.
- **TipTap StarterKit NO bindea `Mod-\\` por default.** `grep "Mod-\\\\"` en `node_modules/@tiptap/` retornó 0 matches. Para validar colisiones de atajos del editor antes de proponer uno nuevo, grep en node_modules del paquete específico es más rápido y autoritativo que context7. Aplicable a cualquier validación de atajos en editores third-party (TipTap, ProseMirror, CodeMirror, Lexical).
- **`npm install` requiere `--legacy-peer-deps`** por conflicto preexistente con `vite-plugin-pwa@1.2.0` que acepta vite ≤7 (proyecto en vite 8.0.8 — deduped y funcional en runtime, solo el peer check falla). Cualquier nueva dep dispara el chequeo y aborta sin el flag. NO lo causa la dep nueva — solo lo expone al re-resolver. Documentar como gotcha del proyecto hasta que vite-plugin-pwa bump su peer.
- **`LayoutStorage` adapter sync ↔ async storage via fire-and-forget.** `getItem` lee del state síncrono de preferences (cache reactiva del onSnapshot), `setItem` parse + `void setPreferences(uid, { ... })` sin await. Si Firestore falla, próximo cambio re-intenta porque el valor en memoria sigue correcto via `onLayoutChanged`. El `debounceSaveMs: 100` nativo del hook ya colapsa decenas de pointermove durante un drag. Patrón reusable para cualquier lib con storage adapter sync API que necesite persistir async.
- **Mapear coexistencia con components ya renderizando en la página antes de meter un layout nuevo.** `BacklinksPanel`/`SimilarNotesPanel` ya estaban en `[noteId]/page.tsx` como sidebar derecha (`lg:flex-row`); con split activo (2 editors + handle + sidebar) el layout colapsaba a <250px por bloque. Mitigación: prop `showSidePanel` al container, forzar `false` durante split, restaurar al cerrar (state del user preservado en `useState`, solo se condiciona visibilidad). Audit pre-código de cualquier feature de layout debe incluir "qué hay aquí ya y cómo coexiste con lo nuevo".
- **TopBar es slot condicional, no universal** — solo se renderiza con `sidebarHidden === true`. Cualquier feature que necesite un botón global discoverable NO puede asumir TopBar — necesita slot alternativo (header del componente principal de la página, sidebar header, CommandPalette entry). F46.6 post-merge resolvió con un segundo callsite en el header del NoteEditor (siempre visible) + entry en CommandPalette. Para features futuras: si el botón es contextual a una página, `headerSlot` del componente principal es el slot más natural.
- **Comunicación cross-tree sibling vía CustomEvent en `window`** cuando Context no aplica. TopBar (descendiente del Layout shell) y SplitPaneLayout (descendiente del Outlet) son siblings en el árbol React — un Provider tendría que vivir en Layout y el SplitPaneLayout publicaría su API via callback (complejidad innecesaria para 1 trigger). Solución MVP: `window.dispatchEvent(new CustomEvent('secondmind:split-open-picker'))` desde TopBar/CommandPalette, `window.addEventListener` en SplitPaneLayout. Listener solo activo cuando SplitPaneLayout está mounted — events emitidos en otras páginas se pierden sin efecto. Patrón simple para 1-3 callers; extraer a Context/hook compartido si emerge un cuarto.
- **`rightStatus` derivado (no boolean isLoading) simplifica render condicional** cuando hay >2 estados async + lifecycle. `idle | loading | ready | not-found` cubre los 4 estados del pane derecho con un solo prop; el consumer hace un switch literal en lugar de combinar múltiples flags. Grace period 500ms post-mount evita falso `not-found` durante hidratación de TinyBase. Auto-cierre 5s post-`not-found` permite al user leer el mensaje + cancelar antes (cerrar manual cancela el timer via cleanup del effect). Aplicable a cualquier hook que retorne estado derivado de async + lifecycle externo.
