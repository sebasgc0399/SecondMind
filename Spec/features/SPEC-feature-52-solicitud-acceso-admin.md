## Objetivo

Reemplazar la gestión 100% manual de la allowlist (seedear `allowlist/{email}` a mano en la Console) por: (1) un **formulario público** donde quien quiere entrar deja su email → cae en una cola estructurada `accessRequests/`; y (2) una ruta **`/admin`** dentro de la app, gateada por uid, donde Sebastián ve la cola y **aprueba/rechaza** sin abrir la Console.

El backstop de seguridad (rules `users/**`, `allowlist/` deny-all, `checkMyAccess`) **no cambió** — recolectar solicitudes ≠ consultar membresía. Esto fue **infraestructura de acceso**, no la apertura de la beta: tras el deploy la beta siguió cerrada (versión `0.4.2`, no `0.5.0`); lo único habilitado fue el ciclo solicitar → aprobar uno a uno sin tocar la Firebase Console.

## Qué se implementó

- **F1 — Cola `accessRequests/` + regla deny-all:** se agregó a `firestore.rules` el bloque `match /accessRequests/{id} { allow read, write: if false; }` (idéntico a `allowlist/`/`userSecrets/`/`rateLimits/`), test de rules que verifica read **y** write denegados para todo contexto cliente (incluida sesión autenticada), y el tipo de dominio. Doc shape `accessRequests/{emailNormalizado}`: `email` (== doc id), `motivo?`, `status: 'pending'|'approved'|'rejected'`, `createdAt`, `processedAt?`. **Sin IP en el doc** (solo hasheada en `rateLimits/`) y **sin índice compuesto** (la cola se ordena en memoria en la CF). Archivos tocados: `firestore.rules`, `firestore.rules.test.ts`, `src/types/accessRequest.ts`.
- **F2 — `submitAccessRequest` (callable pública, anti-spam + no-oráculo):** `onCall` sin `request.auth`, `maxInstances` explícito. Deriva la IP del cliente de `x-forwarded-for` (primer hop, fallback a `request.rawRequest.ip`), la hashea (sha256) y reusa `enforceRateLimit(ipHash, 'access-request', …)`; valida input fail-closed; normaliza el email `trim().toLowerCase()`; dedup en transacción leyendo **`accessRequests/{email}`, nunca `allowlist/`** (no-op si existe, no pisa `createdAt`); responde **uniforme `{ ok: true }`** en todos los casos. Logging vía `sanitizeError`, sin email crudo. Archivos tocados: `src/functions/src/access/submitAccessRequest.ts`, `src/functions/src/index.ts`.
- **F3 — `listAccessRequests` (callable admin-only):** callable autenticada con `secrets: [ADMIN_EMAIL]`; `requireAdmin` (helper nuevo) exige `request.auth` y `token.email` normalizado === `ADMIN_EMAIL` normalizado, si no → `permission-denied`/`unauthenticated`. Admin SDK `where('status','==','pending')`, ordenado por `createdAt` en memoria, limit defensivo. Archivos tocados: `src/functions/src/access/listAccessRequests.ts`, `src/functions/src/lib/requireAdmin.ts`, `src/functions/src/index.ts`.
- **F4 — `processAccessRequest` (callable admin-only, aprobar/rechazar):** callable autenticada `requireAdmin`; input `{ id, action: 'approve'|'reject' }`. En batch: `approve` → `allowlist/{id}` = `{ addedAt: serverTimestamp() }` + request `approved` + `processedAt`; `reject` → request `rejected`, allowlist intacta; doc inexistente → `not-found`. Es el **único** writer a `allowlist/` desde código de app (el Admin SDK bypassa el deny-all; la regla `allowlist/` siguió `if false`). Archivos tocados: `src/functions/src/access/processAccessRequest.ts`, `src/functions/src/index.ts`.
- **F5 — Ruta pública `/solicitar-acceso` + formulario:** wrapper cliente con los 3 `httpsCallable`, ruta fuera del `Layout` (top-level, como `/login`), form con input email + textarea motivo opcional (con cap) → estado de éxito uniforme con copy **truthful** (D6: no promete email), loading como skeleton. Archivos tocados: `src/lib/accessRequests.ts`, `src/app/solicitar-acceso/page.tsx`, `src/components/auth/AccessRequestForm.tsx`, `src/app/router.tsx`.
- **F6 — Ruta `/admin` (lazy, gate por uid) + UI de la cola:** **primer `React.lazy`** del router con `Suspense` fallback skeleton, ruta dentro del `Layout`; gate cosmético `user?.uid !== VITE_ADMIN_UID` → `<Navigate to="/" replace />` (la seguridad real son F3/F4 server-side). Hook `useAccessRequestsQueue` (fetch one-shot `useState`+`useEffect`, fuera de TinyBase/repos — tooling admin efímero). Cada fila con Aprobar/Rechazar → `processAccessRequest` → refetch, empty state + skeleton. Archivos tocados: `src/app/admin/page.tsx`, `src/components/admin/AccessRequestQueue.tsx`, `src/components/admin/AccessRequestRow.tsx`, `src/hooks/useAccessRequestsQueue.ts`, `src/app/router.tsx`.
- **F7 — Gancho de copy: link desde `/login`:** link siempre visible en el footer de `LoginCard` ("¿No tenés acceso? Solicitá una invitación") → `/solicitar-acceso`, cubriendo al que nunca se logueó y al rechazado que rebota a `/login`. Se ajustó `BETA_NO_ACCESS_MESSAGE` para apuntar al link del footer (evita string-matching frágil). Archivos tocados: `src/components/auth/LoginCard.tsx`, `src/lib/authErrors.ts` (+`.test.ts`).

| **D1** | **Cola `accessRequests/` deny-all total + 3 callables**    | Submit, list y process pasan por CF (Admin SDK). `accessRequests/` queda `read, write: if false` igual que `allowlist/`/`userSecrets/`. La colección guarda emails = PII → merece el mismo trato. Modelo uniforme: **ninguna colección con PII es legible desde el cliente**. (Variante rules-first con read condicional por `token.email` descartada: hace PII client-readable + regla frágil ante la aditividad de rules — el gotcha del deny-all top-level.) |

Frena el spam de **fuente única** (el realista para una beta de ~100). Turnstile/reCAPTCHA = escalación documentada si aparece abuso real; YAGNI para un vector que hoy no existe. Escritura directa cliente→Firestore **descartada** (sin freno real al spam, indefendible pre-auth público).                                                                                                       |
| **D3** | **El formulario NO puede ser un oráculo**                  | Respuesta **uniforme siempre**, pase lo que pase (email nuevo / duplicado / ya-allowlisted). Si deduplica, lee **`accessRequests/`, nunca `allowlist/`**, y **nunca** decide el mensaje según membresía. Es A-3 en otra piel.                                                                                                                                                                                                                                   |
| **D4** | **email normalizado = doc id** de `accessRequests/{email}` | `trim().toLowerCase()` (misma convención que `allowlist/`). Dedup natural sin segundo read y sin enumeración. La aprobación escribe `allowlist/{email}` donde `email === id`.                                                                                                                                                                                                                                                                                   |
| **D5** | **`/admin` = solo procesar la cola**                       | Ver pendientes, aprobar (→ allowlist) o rechazar. **Revocar acceso a alguien ya dentro se sigue haciendo a mano en Console** (borrar el doc — raro en beta). | **D6** | **Sin notificaciones por email**                           | Cero infra de envío. El operador revisa `/admin` cuando quiere; el solicitante reintenta el login y entra cuando fue aprobado. ## Rondas de fix

Dos correcciones post-walkthrough antes del merge:

- **`/admin` rebotaba al admin real (`d42c03a`).** Root cause: `AdminPage` evaluaba el gate por uid en el primer render, antes de que su `useAuth` resolviera el `user` async (`onAuthStateChanged` dispara en el próximo tick). Fix: guard `isLoading` antes del gate. El E2E del emulador no lo veía porque prueba las CFs, no el cliente — el bug solo aparece en el walkthrough Playwright real.
- **Estáticos `admin.firestore.FieldValue.*` quedaban `undefined` en el runtime del emulador (`b2a2d7f`).** Fix: imports modulares `firebase-admin/firestore` (`getFirestore`/`FieldValue`/`Timestamp`) en `rateLimit.ts` + las 3 CFs, en vez de los accesos estáticos. Behavior-preserving en prod; gotcha escalado a `cloud-functions-guards.md`.

## Lecciones

- **El E2E del emulador prueba las CFs, no el cliente.** El bug de gate-por-uid de `/admin` (evaluado antes de que auth resolviera) era invisible para el harness del emulador; solo el walkthrough Playwright real lo cazó. Regla general: cualquier gate de ruta en cliente que dependa de estado async (`onAuthStateChanged`, snapshot Firestore) necesita un guard de `isLoading` y verificación visual, no solo cobertura server-side.
- **Imports modulares `firebase-admin/firestore` son emulator-safe; los estáticos `admin.firestore.*` no.** En el runtime del emulador, `admin.firestore.FieldValue`/`.Timestamp` pueden quedar `undefined`. Usar `getFirestore`/`FieldValue`/`Timestamp` de `firebase-admin/firestore`. Canon en `Spec/gotchas/cloud-functions-guards.md`.
- **Un callable público que distingue su respuesta según estado de membresía = oráculo de enumeración.** El no-oráculo se logra con respuesta uniforme + dedup que lee la cola propia (`accessRequests/`) y nunca `allowlist/`. Principio ya canonizado en F51 ("callable público que revela estado = oráculo"); F52 lo reaplicó. La derivación de IP por `x-forwarded-for` resultó trivial, sin gotcha aparte.
- **`enforceRateLimit(clave, …)` no está acoplado a uid** — el primer arg es un string arbitrario, así que es reutilizable pre-auth pasando un hash de IP como clave. Patrón válido para cualquier endpoint público que necesite freno de spam de fuente única.
- **Toda colección con PII (emails) merece deny-all client-side + acceso exclusivo vía Admin SDK.** Exponerla con read condicional por `token.email` la haría client-readable y dependería de una regla frágil ante la aditividad de Firestore rules. El modelo uniforme (`allowlist/`, `userSecrets/`, `accessRequests/`, `rateLimits/` todas deny-all) es más simple de razonar y más difícil de romper accidentalmente.
