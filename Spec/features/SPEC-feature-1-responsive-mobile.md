## Objetivo

El usuario abre SecondMind desde su celular (PWA o APK Capacitor) y tiene una experiencia nativa: bottom nav para navegar entre secciones primarias, FAB para Quick Capture, content area que no se desborda ni requiere scroll horizontal involuntario, tap targets de 44px mínimo, y safe-area insets respetados (notch, gesture bar, status bar). En tablet la sidebar se colapsa a iconos de 64px con hamburger que abre un drawer overlay. En desktop el layout previo se preserva sin regresión.

Sin esto, el dev server fallaba con `Component auth has not been registered yet` porque el optimizer Vite recogía paquetes `@firebase/*` desde `extension/node_modules/` (Chrome Extension workspace separado) y duplicaba el component registry. Bug existía desde Fase 5.2; descubierto en el audit inicial con Playwright. Commit aislado antes de F1 para trazabilidad. Archivos tocados: `vite.config.ts`.

## Decisiones clave

1. **Una sola surface de drawer (`NavigationDrawer`) reusada en mobile + tablet.** No hay estado intermedio "sidebar collapsed + expanded overlay push" — el hamburger en tablet abre el mismo drawer full-screen que mobile. Simplificación YAGNI.
2. **`SidebarContent` extraído como componente exportado y reusado en `NavigationDrawer`.** Cero duplicación de JSX entre Sidebar desktop y drawer mobile/tablet. Acepta `onNavigate?` para que cada NavLink cierre el drawer al clickear.
3. **Safe-area insets aplicados granular (top en MobileHeader, bottom en BottomNav/FAB), NO globales en body.** Solo `padding-left/right` van al body porque son uniformes; top y bottom van al elemento que toca el edge — duplicar padding genera gaps visibles.
4. **Tap targets via `<label>` wrapper sobre input/control pequeño** (`<input h-4 w-4>` dentro de `<label h-11 w-11>` con negative margin compensador). Mantiene el visual original y da 44×44 de hit area sin tocar el styling del control. Aplicable a cualquier Radix/base-ui input chico.
5. **`useBreakpoint()` con fallback `'tablet'` en server snapshot** (cuando ambas queries devuelven false). Evita flash en primer render sin crear mismatch de hidratación. En WebView Capacitor la hidratación es sincrónica → no hay flash visible.
6. **MobileHeader sin back button dinámico.** Los back links inline en cada página de detalle ya funcionan; un registry de rutas padre era complejidad innecesaria. YAGNI.
7. **HabitGrid mantuvo `<table>` con `position: sticky` en `th/td:first-child`**, no migró a CSS grid. La table existente funcionó tal cual con menos cambios.
8. **GraphFilters preserva accordion con `useState`**, no migró a `<details>/<summary>` nativo. Componente existente ya funcional; refactorizar no agregaba valor. Solo ajuste de paddings responsive.
9. **FAB cubre `QuickCaptureButton` del dashboard en mobile** (`hidden md:block` + FAB montado solo si `isMobile`). Misma acción, distintos affordances por viewport — no se duplican en pantalla.

## Lecciones

- **El SW cache del WebView Capacitor persiste entre reinstalaciones del APK.** Al reinstalar con `adb install -r` o transferencia manual, el `vite-plugin-pwa` Service Worker retiene el bundle del install anterior y lo sirve al abrir — el user ve la versión vieja en la primera apertura. `registerType: 'autoUpdate'` resuelve en reloads subsiguientes pero la primera apertura engaña. Resolución: Ajustes → Apps → SecondMind → Almacenamiento → Borrar caché, o desinstalar completo antes de cada install para testing E2E confiable.
- **Hit area via `<label>` wrapper sobre control pequeño es el patrón canónico** para cualquier checkbox/radio/switch chico (Radix, base-ui, native). El label recibe el click del browser y togglea el input por herencia, así que no se necesita JS extra. Negative margin (`-my-2 -ml-2`) compensa el extra height sin desbalancear el container.
- **`useSyncExternalStore` con server snapshot estable no causa flash visible en SPA / Capacitor WebView.** Sin SSR real, el primer paint usa el snapshot real (post-mount) inmediatamente. El fallback en server snapshot solo importa si rendereás en Node — para SPA puro un valor sensato (no `undefined`) basta para evitar mismatch defensivo.
- **Tablet puede reusar la surface de drawer mobile en lugar de tener su propio "sidebar overlay push"** cuando el real estate es escaso. Una sola superficie + transición consistente vale más que dos estados de sidebar (collapsed 64 + expanded overlay) — menos código, menos casos edge, mismo UX.
- **Safe-area insets se aplican al elemento que toca el edge, NO al body globalmente.** `padding-left/right` van bien al body (uniforme en toda la app), pero `padding-top` global duplica con el header sticky y `padding-bottom` global duplica con el nav fijo, dejando gaps visibles. Aplicar `var(--sai-top)` en MobileHeader y `var(--sai-bottom)` en BottomNav/FAB es el patrón correcto.
