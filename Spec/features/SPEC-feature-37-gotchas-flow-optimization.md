## Objetivo

Reducir el costo en tokens de consultar gotchas en el SDD. Pre-F37 `Spec/ESTADO-ACTUAL.md` mezclaba **índice de features** (función liviana) con **canon completo de gotchas por dominio** (función pesada que crecía monotónicamente — 351 líneas, ~226 corpus). Cualquier agente que necesitaba "el gotcha de TipTap sobre marks" cargaba las 351 líneas, filtraba mentalmente, y descartaba el 90% del contenido.

Post-F37: (a) los gotchas viven en `Spec/gotchas/<dominio>.md` con índice de 1 línea en ESTADO-ACTUAL apuntando al canónico; (b) skill local on-demand `gotchas-search` con BM25 sirve queries por keyword/dominio; (c) `CLAUDE.md` actualizado con 5 pointers a la nueva ubicación — sin imponer flujo de subagentes (Fase C original diferida hasta evidencia de fricción manual).

`Spec/gotchas/<dominio>.md` es nivel **lateral** a ESTADO-ACTUAL — NO un nivel nuevo en la jerarquía CLAUDE.md → ESTADO-ACTUAL → SPEC.

## Lecciones

- **BM25 stdlib es suficiente para corpus < 500 entries; threshold tokenize `> 1` (no `> 2`) para preservar tokens técnicos cortos** (`ui`, `css`, `cf`, `h1`, `js`). El patrón canónico de `ui-ux-pro-max` usa `len(w) > 2` apropiado para producto/diseño donde tokens cortos son ruido. Para corpus de gotchas técnicos, ese threshold filtra términos críticos del dominio. Aplicable a cualquier futura skill BM25 sobre dominio con vocabulario técnico cortos.
- **`json.dump(..., ensure_ascii=False, sort_keys=True)` obligatorio para artifacts de skills bilingues regenerados por hook.** Sin `ensure_ascii=False` los acentos y `ñ` quedan como `ñ` haciendo diffs ilegibles para code review. `sort_keys=True` garantiza idempotencia byte-a-byte (mismo input → mismo output) — sin esto, el reindex regenera output con orden no-determinístico de claves y polluta git con churn artificial. Aplicable a cualquier futuro corpus.json o artifact serializado bajo hook automation.
- **Atomic write con `os.replace(tmp, final)` para artifacts regenerados bajo race con MultiEdit.** MultiEdit toca N archivos secuencialmente, hook PostToolUse corre N veces seguidas. Sin atomic write, una invocación de `search.py` concurrente puede leer JSON parcial mid-write. Patrón canónico: `tempfile.mkstemp` en mismo directorio + `os.replace`. Aplicable a cualquier futuro artifact regenerado por hook bajo carga de Edit batch.
- **`$USERPROFILE` (no `~`) y pattern OR `forward|backslash` en `case` glob de hooks PostToolUse en Windows.** Tilde no expande dentro de `case` heredocs invocados desde Claude Code; `$USERPROFILE` sí. `$CLAUDE_TOOL_INPUT_FILE_PATH` puede venir con backslashes en Windows — pattern `*"Spec/gotchas/"*".md"` solo matchea forward; OR con `*"Spec\\gotchas\\"*".md"` cubre ambos. Aplicable a cualquier futuro hook con path matcher Windows-aware.
- **`.claude/settings.json` se cachea por sesión de Claude Code.** Cambios al archivo NO se aplican a la sesión activa — el hook nuevo empieza a disparar en la próxima sesión. Validar el comando manualmente con bash directo (replicando el `case esac` con `$CLAUDE_TOOL_INPUT_FILE_PATH` simulado) antes de cerrar el commit, porque smoke test in-session del hook recién añadido es imposible. Aplicable a cualquier futuro F sobre `.claude/settings.json`.
- **`sys.stdout.reconfigure(encoding="utf-8")` obligatorio en scripts Python invocados por hook en Windows.** Default cp1252 codepage choca con caracteres Unicode (`→`, `—`, acentos) presentes en bodies de gotchas técnicos: `UnicodeEncodeError: 'charmap' codec can't encode character '→'`. El fix tiene que ser programático (no env var `PYTHONIOENCODING=utf-8`) porque el hook se invoca sin shell ENV setup. Aplicable a cualquier futura skill Python con output Unicode bajo hook automation Windows.
