## Objetivo

Cerrar los 2 hallazgos pre-existentes detectados en la auditoría de F10 (capa de repos), ambos al lifecycle de los stores TinyBase: (a) cross-user data leak <100ms en user switch porque los stores singleton no se limpiaban antes de hidratar al user nuevo; (b) `isInitializing` con timer arbitrario de 200ms en 8 consumidores, no sincronizado con el estado real del persister. F10 centralizó los **writes** vía `createFirestoreRepo`; F11 centraliza el **gate de lectura** vía `useStoreHydration`. F4 era spike adicional (retry+backoff para auth errors) con pre-bias hacia descartar.

## Qué se implementó

- **F1 — `delTable` pre-init y en cleanup en useStoreInit:** las 7 tablas TinyBase se vacían antes del `Promise.all(startAutoLoad)` con el `userId` nuevo, y en el cleanup post-`destroy()`. Cierra la ventana <100ms del leak. delTable con persister snapshot-based NO borra Firestore (`Object.entries({}).map() → Promise.all([]) → cero setDoc` — confirmado por auditoría). Orden crítico en cleanup: `destroy()` antes de `delTable`, sino el snapshot listener in-flight repuebla la tabla vacía. Archivos tocados: `src/hooks/useStoreInit.ts`.
- **F2 — Signal real `useStoreHydration` via Context + Provider:** `useStoreInit(userId)` ahora retorna `{ isHydrating: boolean }` derivado de `hydratedUserId !== userId` (patrón derivado evita `setState` dentro del effect). Anti-StrictMode: `currentUserIdRef` filtra `.then` resolutions de userId obsoleto en double-mount de dev. Context con default value `{ isHydrating: true }` (skeleton-safe sin Provider). Provider montado en `layout.tsx` envolviendo `<CommandPaletteProvider>` (queda arriba en el árbol de CommandPalette → useHybridSearch → useNoteSearch). Hook + Context separados del Provider en archivos distintos para no romper `react-refresh/only-export-components`. Archivos tocados: `src/hooks/useStoreHydration.ts` (nuevo), `src/hooks/StoreHydrationProvider.tsx` (nuevo), `src/hooks/useStoreInit.ts`, `src/app/layout.tsx`.
- **F3 — Migración de 9 consumidores al signal real:** patrón mecánico de borrar `useState(true) + setTimeout(200) + INIT_GRACE_MS` y delegar a `useStoreHydration()`, aliaseando `isHydrating` como `isInitializing` en el return (compat con callers, misma API). 7 hooks (useTasks, useProjects, useObjectives, useHabits, useInbox, useNoteSearch, useDailyDigest) + 2 componentes (RecentNotesCard, DailyDigest — este último también eliminó su `showSkeleton` redundante). useNoteSearch conservó el rebuild de Orama via `addTableListener` (reactivo). useHybridSearch hereda automático via useNoteSearch. usePendingInboxCount NO migró (retorna `number` directo, tolera 0 inicial sin flicker). `grep INIT_GRACE_MS src/` = 0. Archivos tocados: 7 hooks + 2 componentes.
- **F3b — Reemplazo del grace de 1500ms en projects/[projectId]:** workaround viejo del `isInitializing` 200ms se reemplaza por condición determinística `!isHydrating && !project`. Borra `redirectGraceExpired` state + setTimeout(1500ms). Skeleton condicional + redirect ahora dependen del signal real. Archivos tocados: `src/app/projects/[projectId]/page.tsx`.
- **F4 — Descartado:** spike de retry+backoff para auth errors no se implementa. Pre-bias confirmado por auditoría: Firebase Auth refresca tokens cada ~1h y `onSnapshot` se re-arma solo cuando el SDK detecta el refresh. Un retry custom duplicaría trabajo del SDK y arriesga loops en logout legítimo. Sin cambios a `src/lib/tinybase.ts`. E2E del escenario "token expiry mid-sesión" diferido (requiere esperar 1h+1min real); riesgo bajo según la auditoría.
- **F5 — E2E parcial:** validado runtime el flow signOut → login → tarea persiste, lo que confirmó F1 cleanup vacía memoria + F2 reinit hidrata desde Firestore correctamente. Cross-user real con segundo Google account y skeleton timing con Network throttle CDP diferidos por setup (popup OAuth requiere interacción manual; Network throttle no expuesto en Playwright MCP).

## Lecciones

- **delTable con custom persister snapshot-based NO borra Firestore.** El callback `setPersisted` itera `Object.entries(rows)` del snapshot. Tabla vacía → `Promise.all([])` → cero `setDoc`. TinyBase v8 no hace diff-detection de "esta row desapareció, borrarla". Aplica a cualquier custom persister con la misma forma — leer el snapshot completo y escribir lo que ve, no diffs.
- **State derivado vence a setState en effect** para gates basados en una identidad externa (userId). En vez de `setIsHydrating(true)` al cambiar userId (que dispara cascading renders + viola `react-hooks/set-state-in-effect`), mantener `[hydratedUserId, setHydratedUserId]` y derivar `isHydrating = userId !== hydratedUserId`. El único setState es cuando termina la hidratación. Patrón generalizable para cualquier "estado de carga sincronizado con un input externo".
- **StrictMode + Promise.all en useEffect requieren ref guard.** Double-mount en dev cancela y reinicia el Promise.all; sin guard el `.then` del primero puede resolver tarde y flippear estado del segundo. `currentUserIdRef.current = userId` al inicio del effect + chequeo `currentUserIdRef.current === userId` en el `.then` filtra resolutions obsoletas. Aplica a cualquier async work disparado por dependency change con setState en el resultado.
- **Hook + Provider en mismo archivo `.tsx` rompe `react-refresh/only-export-components`.** La regla solo exime archivos `.ts` puros (sin JSX). Patrón canónico: archivo `.ts` con Context + interface + hook (sin JSX), archivo `.tsx` separado que importa el Context y exporta solo el Provider component. Cero disable comments.
- **Custom persister no necesita signal "ready" explícito** si los consumidores leen reactivo del store (vía useTable / useCell). El `Promise.all(startAutoLoad)` es la transición natural; expose ese promise via Context y los consumidores skeletean sin código de timing local.
- **Workarounds de timing arbitrario tienden a duplicarse.** El timer de 200ms en useTasks llevó a un grace de 1500ms en projects/[projectId] como compensación. Cuando se reemplaza el origen por un signal real, ambos workarounds caen automáticamente. Buscar este patrón al introducir cualquier signal nuevo: el primer call site no es el único.
- **Embeddings cache + module-level singletons necesitan `cachedUid` guard explícito.** El cache de embeddings (`src/lib/embeddings.ts`) ya tenía esto bien (auditoría confirmada). Cualquier cache module-level que sobreviva entre user switches debe versarse por userId + invalidarse en signOut, no asumir que el módulo se re-importa.
