## Objetivo

El usuario busca en `/notes` y encuentra notas por significado, no solo por palabras exactas. La query "inteligencia artificial" surfacea notas sobre "Claude Mythos" o "Claude Code Security" aunque la frase exacta no aparezca en el texto. Keyword sigue instant para el caso dominante; semántico es aditivo (no rankea mixto, no bloquea).

## Qué se implementó

- **F1 — Cloud Function `embedQuery` callable:** `onCall` v2 (`firebase-functions/v2/https`), input `{ text }` validado no vacío y ≤ 500 chars, output `{ vector: number[] }` de 1536 dims. Modelo `text-embedding-3-small` (mismo que `generateEmbedding` para que los vectores sean comparables). Secret `OPENAI_API_KEY` reutilizada. Auth requerido (`request.auth.uid`, sino `HttpsError('unauthenticated')`). Errores internos via `HttpsError('internal', ...)` + `logger.error` para telemetría. `timeoutSeconds: 10`, región `us-central1`. **Primera CF callable del proyecto** — las 3 previas son event-driven. Archivos tocados: `src/functions/src/search/embedQuery.ts` (nuevo), `src/functions/src/index.ts`.
- **F2 — Cache module-level de embeddings:** `getEmbeddingsCache(uid)` con load lazy on-demand y dedup de fetches concurrentes via `fetchPromise`, reusa si el `uid` no cambió. `invalidateEmbeddingsCache()` para reset (llamado en logout — defensivo contra login con otra cuenta en la misma sesión). `updateEmbeddingInCache(noteId, vector)` para updates puntuales. Refactor de `useSimilarNotes` para consumir cache compartido (antes cada navegación entre notas re-fetcheaba todos los embeddings — el `cacheRef` local sólo vivía por montaje del panel). Archivos tocados: `src/lib/embeddings.ts`, `src/hooks/useSimilarNotes.ts`, `src/hooks/useAuth.ts`.
- **F3 — Hook `useHybridSearch`:** `getFunctions(app, 'us-central1')` exportado desde `src/lib/firebase.ts` (primer uso de Firebase Functions cliente). `embedQueryText` via `httpsCallable<{text}, {vector}>(functions, 'embedQuery')` con referencia creada module-level (una sola vez). Compone `useNoteSearch()` keyword sin refactorizar — reutiliza la lógica sync + filter de archivadas tal cual. Debounce vanilla 500ms (`useRef<number>` + `setTimeout`, cleanup en effect return). Guards: `!user || trimmed.length < 3` → `shouldSearch = false`. **Pipeline estricto en orden**: `filter isArchived → exclude keyword IDs → sort score desc → slice(0, 5)` — si se slicea antes de excluir, con 5 keyword queda `semanticResults` vacío. Race handling por snapshot del query (`frozenQuery !== query.trim()` descarta resultados obsoletos; sin `AbortController` porque Firebase callable no lo expone). Degradación graceful con `try/catch` → `semanticResults: []` sin error visible. `setIsLoadingSemantic(true)` DENTRO del `setTimeout` callback (respeta `react-hooks/set-state-in-effect` y el skeleton aparece cuando arranca la llamada real, no durante el debounce). Archivos tocados: `src/lib/firebase.ts`, `src/lib/embeddings.ts`, `src/hooks/useHybridSearch.ts` (nuevo).
- **F4 — UI sección semántica en `/notes`:** `NoteCard` con prop opcional `semanticScore?: number` que renderiza badge violet con ícono `Sparkles` + porcentaje (`bg-violet-500/10 text-violet-500 rounded-md px-1.5 py-0.5 text-xs`); consumers existentes (Command Palette, dashboard) no cambian. Page reemplaza `useNoteSearch` por `useHybridSearch`. `SemanticSection` aparece cuando hay query activa con heading "Semánticamente similares"; estados skeleton mientras `isLoadingSemantic`, resultados con score, o sección oculta si 0 resultados sin loading. Caso "sin keyword pero con semánticos" sube como resultado principal con leyenda "No hay coincidencias exactas, pero estas notas son temáticamente similares." Caso "sin keyword y sin semánticos": "Sin resultados." en dashed border. Archivos tocados: `src/components/editor/NoteCard.tsx`, `src/app/notes/page.tsx`.
- **F5 — Calibración empírica del threshold:** `SEMANTIC_THRESHOLD` bajado de 0.45 (SPEC original) a 0.30. Con `text-embedding-3-small` + notas cortas en español la cosine similarity satura en 0.15–0.45 (no 0.7–0.95 como `ada-002`). Top match genuino "Claude Mythos IA" para "inteligencia artificial" daba 0.427 — el semántico no mostraba nada útil con el threshold del SPEC. Con 0.30 los 3 matches legítimos de IA pasan (0.43, 0.33, 0.27) y el ruido de metodologías dispares (0.23) y templates (0.25) queda descartado. Subir a 0.35 es 1-line change si futuras notas con otro estilo muestran ruido. Archivos tocados: `src/hooks/useHybridSearch.ts`.

## Decisiones clave

1. **Callable `onCall` v2 en vez de HTTP `onRequest`** — auth automática via Firebase Auth token, serialización JSON, errores via `HttpsError` mapeados al cliente. Menos código manual que `onRequest` + CORS + manual token validation.
2. **Cache module-level compartido entre hooks** — deduplica fetches de `users/{uid}/embeddings` cuando `SimilarNotesPanel` y `useHybridSearch` se activan en la misma sesión. Se invalida en logout.
3. **Secciones separadas (no RRF/fusion ranking)** — del SPEC. Evita "resultado shuffling" cuando llegan los semánticos 500ms después de los keyword.
4. **Threshold 0.30 empírico** (F5). Calibrado con corpus real, no con un número del SPEC.
5. **Debounce vanilla 500ms** — sin librería, 15 líneas, `setTimeout + useRef`. Se siente responsivo sin hammerear la CF.
6. **Pipeline estricto**: `filter isArchived → exclude keyword IDs → sort → slice`. Si se invierte slice y exclude, semantic queda vacío cuando keyword devuelve 5+ resultados.
7. **Race handling por snapshot del query** — `frozenQuery !== query.trim()` descarta resultados obsoletos. Sin `AbortController` porque Firebase callable no lo expone.
8. **`setIsLoadingSemantic(true)` dentro del `setTimeout` callback, no síncrono en effect** — respeta `react-hooks/set-state-in-effect` y alinea UX: skeleton al arrancar la llamada real, no durante el debounce.
9. **Degradación graceful** — `try/catch` + `semanticResults: []` en error. Sin toast: el usuario ve keyword results y no percibe diferencia si la CF falla.
10. **`semanticScore` como prop opcional en `NoteCard`** — no rompe consumers existentes (Command Palette, Dashboard, etc.).

## Lecciones

- **Firebase Functions v2 callable signature: `(request) => { request.auth?.uid; request.data }` con errores via `HttpsError`.** `context.auth` y argumentos posicionales son v1 — cualquier guía o snippet pre-2023 va a estar stale. Si se mezcla la sintaxis v1 con runtime v2, el handler compila pero `request.auth` sigue undefined y el guard de auth siempre falla open o cierra sin razón aparente.
- **`text-embedding-3-small` con textos cortos en español satura cosine similarity en 0.15–0.45**, no en 0.7–0.95 como `ada-002`. Cualquier threshold del SPEC original basado en intuición de modelos viejos hay que **recalibrar empíricamente con corpus real**, no usar el número del paper ni de docs OpenAI ni del SPEC. Aplica a cualquier migración entre modelos de embeddings (incluso entre versiones del mismo proveedor).
- **`firebase.json.runtime` manda sobre `package.json.engines.node` en deploy.** Si hay mismatch (ej. `firebase.json: nodejs20` y `package.json: 22`), el runtime efectivo es el de `firebase.json` y nadie te avisa. Alinearlos siempre o documentar el desvío explícitamente; cualquier "fix" silencioso en uno de los dos cambia el runtime de producción sin otras señales.
- **`useRef` dentro de un hook NO sobrevive al desmontaje del componente que lo usa.** Cualquier cache que necesite persistir entre montajes (embeddings entre navegaciones, query results compartidos entre paneles, auth state entre routes) debe vivir **module-level** (variable top-level del módulo) o en context — usar `useRef` para cache compartido es un bug que sólo se manifiesta cuando el usuario navega y vuelve.
- **Pipeline order matters: `filter → exclude → sort → slice`, en ese orden estricto.** Cualquier reordenamiento (slice antes de exclude, sort antes de filter) genera bugs sutiles que sólo se ven con corpus grande. Si la lista derivada queda inesperadamente vacía o duplicada, recheck primero el orden de las operaciones del pipeline antes de buscar el bug en otro lado.
