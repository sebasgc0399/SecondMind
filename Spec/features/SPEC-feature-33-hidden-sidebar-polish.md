## Objetivo

Cerrar las dos deudas residuales del out-of-scope de F32 sin alterar `UserPreferences`, schema Firestore ni Cloud Functions: (1) exit-anim simétrico del swap sidebar↔TopBar (hoy solo el entrante anima, el saliente desmonta instantáneo), (2) entrypoint mouse-only en TopBar para mostrar el sidebar oculto (hoy solo Cmd+B + Settings + Command Palette). 100% client-side; cero migración de datos.

## Qué se implementó

- **F33.1 — Exit-anim simétrico del swap sidebar↔TopBar (`src/hooks/useMountedTransition.ts`, `src/hooks/useMountedTransition.test.ts`, `src/components/layout/Sidebar.tsx`, `src/components/layout/TopBar.tsx`, `src/app/layout.tsx`):** hook nuevo `useMountedTransition(visible: boolean, durationMs: number): { shouldRender: boolean; isExiting: boolean }` que retarda el unmount durationMs para permitir `animate-out slide-out-to-X` antes de desmontar. State interno combinado `{ shouldRender, isExiting, prevVisible }` con detección de cambios DURANTE render (`if (state.prevVisible !== visible)`) — patrón canonizado en `useExpandThenCollapse`. Skip-initial gratis (en mount inicial `prevVisible === visible`, la rama de update no corre) y pre-paint timing sin `useLayoutEffect` (el setState durante render ocurre en el mismo ciclo del paint). 6 tests unitarios pasan: arranque visible=true, arranque visible=false skip-initial, flip true→false retarda durationMs, flip false→true monta inmediato, toggle rápido <durationMs cancela timer, cleanup en unmount cancela timer pendiente. Sidebar y TopBar suman prop opcional `animateExit?: boolean`; cuando `true` aplica `animate-out slide-out-to-left fill-mode-forwards duration-200` (Sidebar) o `animate-out slide-out-to-top fill-mode-forwards duration-200` (TopBar) vía `cn()`. `fill-mode-forwards` obligatorio para evitar revert 1 frame pre-unmount. Layout consume `useMountedTransition(showSidebar, 200)` y `useMountedTransition(showTopBar, 200)`, render condicional sobre `shouldRender`, pasa `animateEntry={animateLayoutSwap && !transition.isExiting}` (mutua exclusión G1) y `animateExit={transition.isExiting}`. `showSidebar`/`showTopBar` movidos arriba del early return (`if (isLoading) return ...`) para compatibilidad con rules-of-hooks. La lógica F32.3 de `animateLayoutSwap` (toggle-only entry-anim con `useLayoutEffect` + `isInitialMount` ref) queda intacta — F33 suma el saliente sin reemplazar el entrante.

- **F33.2 — Botón "Mostrar menú" en TopBar (`src/components/layout/TopBar.tsx`):** botón explícito antes del logo, icono `PanelLeftOpen` de lucide-react (ya importado en `CommandPalette.tsx` post-F32.1), `aria-label="Mostrar menú"` + `title="Mostrar menú"` para descubribilidad mouse + accesibilidad. Click → `setPreferences(user.uid, { sidebarHidden: false })` con guard `if (!user) return` defensivo. Wrapper `<div className="flex items-center gap-1">` agrupa botón + Link en el cluster izquierdo del header (justify-between distribuye con el cluster derecho intacto). Sin shortcut kbd visible — el botón es entrypoint mouse-only complementario a Cmd+B. Solo monta en desktop hidden mode (vive dentro del TopBar que ya gatea por `breakpoint === 'desktop' && sidebarHiddenEffective`).

## Lecciones

- **Patrón `setState durante render` con state combinado `{ value, key }` para detectar cambios de prop en hooks reutilizables.** Cuando un hook necesita reaccionar a un cambio de input prop (visible flippea, triggerKey cambia, breakpoint flippea, etc.) el reflejo común es `useEffect` con `[input]` en deps + `useRef<boolean>(true)` para skipear el primer effect post-mount. Pero ese patrón requiere ref auxiliar, corre POST-paint (problemas de timing si el setState es source de una clase de animación pre-paint), y a menudo termina con `eslint-disable react-hooks/set-state-in-effect`. La alternativa idiomática React 19: state combinado `{ ...derived, prevInput }` + check durante render `if (state.prevInput !== input)` + setState durante render para sincronizar. React re-ejecuta el render con state nuevo SIN commit intermedio (ver [React docs](https://react.dev/reference/react/useState#storing-information-from-previous-renders)). Skip-initial gratis (en mount `state.prevInput === input` por construcción), pre-paint timing gratis (mismo ciclo del render que dispara el paint), sin ref auxiliar, sin `eslint-disable`. Aplicable a cualquier futuro hook reutilizable que detecte cambios de prop. Patrones vivos: `src/hooks/useExpandThenCollapse.ts:19-38` (visibility toggle con timeout), `src/hooks/useMountedTransition.ts` (delay unmount con timeout). Escalado a `Spec/ESTADO-ACTUAL.md` (sección hooks).

- **`fill-mode-forwards` obligatorio en `animate-out` que combina con unmount post-anim.** `tailwindcss-animate` (base de `tw-animate-css`) no aplica `animation-fill-mode: forwards` por default — el componente revierte a estado inicial al terminar la anim, lo que produce un flash visual de 1 frame antes del unmount. La utility `fill-mode-forwards` de tw-animate-css resuelve. Aplicable a cualquier futura `animate-out` aplicada vía `cn()` con boolean (no `data-*` gating de base-ui que tiene su propio fill-mode handling). F33.1 fue el primer caso del repo de `animate-out` aplicado vía `cn()` con boolean — antes solo `alert-dialog.tsx` lo usaba con `data-closed:` de base-ui. Escalado a `Spec/ESTADO-ACTUAL.md` (sección animations / chrome global).

- **Mutua exclusión `animateEntry`/`animateExit` por construcción del consumer.** Cuando un componente acepta ambas props animation-control desde el exterior, el consumer es responsable de garantizar la mutua exclusión — no el componente. Patrón: `animateEntry={otherCondition && !transition.isExiting}` + `animateExit={transition.isExiting}` en el padre. Sin el guard `&& !isExiting`, la combinación `animate-in slide-in-from-X` + `animate-out slide-out-to-X` produce comportamiento CSS undefined. Plan agent (step 2 SDD) detectó este gotcha proactivamente. Aplicable a cualquier futuro componente con animation entry+exit controladas por props booleanas. Validado E2E con sampling DOM cada 30ms en Tests 1 y 2.

- **Step 2 SDD justifica re-evaluar directivas previas del usuario cuando el audit ilumina alternativas no consideradas.** El user pidió originalmente Opción A (`useRef + isInitialMount`) para `useMountedTransition` por paridad con F32.3. Phase 1 audit (Agent #1 mapeo de hooks) descubrió `useExpandThenCollapse` como hook canónico con patrón distinto (Opción B `setState durante render`). Plan agent (Phase 2) recomendó firmemente Opción B con 4 razones técnicas. La directiva original asumía paridad con F32.3 (workaround inline en consumer) sin saber del patrón canónico. Resolución: AskUserQuestion presentando ambas opciones con código preview lado-a-lado. User confirmó Opción B con la nueva información. Lección: el SPEC autoriza al implementador a recomendar alternativas cuando el audit revela mejor patrón; el AskUserQuestion preserva la decisión final del user sin imponer la recomendación silenciosamente. Patrón replicable para futuros conflictos directiva-vs-audit en step 2 SDD.

- **Sampling DOM cada 30ms con `setInterval` + Promise valida timing fino más confiable que screenshots.** Para validar coexistencia simultánea de `animate-in` + `animate-out` durante 200ms, screenshots de Playwright (~50-100ms latencia) no capturan el frame medio confiablemente. `browser_evaluate` con `() => new Promise(resolve => { ... setInterval(tick, 30) ... })` da timeline preciso, push N samples a un array, devuelve al main para análisis. Patrón ya documentado en F32 lección 4 — F33 lo reutiliza para Tests 1, 2, 3, 4, 5.

## Out of scope (deuda explícita)

- **Layout shift horizontal instantáneo en escenario B (hidden → visible).** Cuando el sidebar entrante monta al t=0 con `w-64`, el column flex es empujado a la derecha en 1 frame antes de que el `slide-in-from-left` arranque. Mismo trade-off que ya existía post-F32.3. Resolverlo violaría D2 simétricamente (`position: absolute` para el entrante con bounds capture). F34 puede revisitar si UX lo pide.
- **Hover-edge auto-show del sidebar.** Descartado en D5 por complejidad de state mgmt. Si surge demanda concreta de "modo dock" tipo macOS, escalar a F34.
- **Animar el width del main durante el exit (smooth pop).** Requiere `position: absolute` durante exit + capture de bounds (D2 opción A). Aceptado v1.
- **`getModKey()` util cross-platform** para renderizar `⌘B` en mac y `Ctrl B` en win/linux dinámicamente. Sigue postergado de F31/F32 — el botón nuevo no muestra shortcut, F33 no fuerza la decisión.
- **Cleanup explícito del hint en signOut.** Sigue postergado de F32. F33 no toca `preferences.ts` ni el hint.
- **Race del hint stale cross-device.** Sigue aceptado v1 (documentado en F32). F33 no afecta.
- **Refactor a `useQuickActions()` extraído** del CommandPalette. Sigue postergado de F32.
- **Toggle rápido durante exit re-anima `animate-in` sobre componente ya visible.** Si el user dispara segundo toggle <200ms del primero, el componente que mantiene `shouldRender=true` recibe nueva clase `animate-in` cuando `animateLayoutSwap` se re-dispara. La animación CSS arranca desde keyframe initial (translateX -100%) y desliza a 0 — produce un blip visual sutil cuando el componente ya estaba a 0. Validado E2E (Test 5): comportamiento funcionalmente correcto (estado final correcto), pero la transición intermedia muestra el slide-in completo. Mitigación posible: condicionar `animateEntry` también en `transition.shouldRender` flippeo de false→true (no aplicar si ya estaba mounted). Defer hasta feedback de UX.
