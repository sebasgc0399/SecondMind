## Objetivo

Cerrar el silent data loss en los `create*` del factory `createFirestoreRepo`. Pre-F30, `update` y `remove` ya pasaban por queue (post-F29) con backoff + sign-out guard, pero `create` seguía haciendo `await setDoc` directo: si la red fallaba durante el create, la row local en TinyBase quedaba escrita (porque `setRow` corre sync ANTES del `setDoc`) pero el doc en Firestore nunca llegaba — próxima hidratación, `onSnapshot` borraba la row local con un snapshot stale.

El bloqueo arquitectónico de F29 era el contrato: encolar `setDoc` rompía `Promise<string>` que asumen los callers (`const id = await createNote(); navigate('/notes/' + id);`). F30 lo resuelve generando UUID local sync, aplicando `setRow` sync, encolando `setDoc(merge:true)` con backoff y retornando el id INMEDIATAMENTE. La página destino tolera la ventana "row en TinyBase pero doc no en Firestore" cross-checkeando contra el queue de creates.

## Qué se implementó

- **F30.1 — Singletons de creates:** 4 nuevos `SaveQueue<Row>` (`saveNotesCreatesQueue` con intersección `& { content?: string }` para `createFromInbox`, `saveTasksCreatesQueue`, `saveProjectsCreatesQueue`, `saveObjectivesCreatesQueue`). Array `createsQueues` por simetría con `metaQueues` y `allQueues` extendido a 11 (1 content + 6 metas + 4 creates) — los hooks/UI agnósticos de F29.5 los recogen automáticamente. +1 unit test cubriendo payload con field extra-schema. Archivos tocados: `src/lib/saveQueue.ts`, `src/lib/saveQueue.test.ts`.
- **F30.2 — Factory `createsQueue?` + saveContent migración:** `RepoConfig<Row>` acepta `createsQueue?: SaveQueue<Row>` opcional; con queue, `create` encola `setDoc(merge:true)` con backoff + sign-out guard (paridad con G1 de update/remove) y retorna el id sync. Sin queue, await directo (retro-compat). En el mismo commit, `notesRepo.saveContent` migró de `updateDoc` a `setDoc(merge:true)` (D9): `updateDoc` fallaba con `not-found` si el doc aún no existía en server durante create-pending. +3 unit tests. Archivos tocados: `src/infra/repos/baseRepo.ts`, `src/infra/repos/baseRepo.test.ts`, `src/infra/repos/notesRepo.ts`.
- **F30.3 — Wiring 4 repos:** inyección mecánica del `createsQueue` correspondiente en `notesRepo`, `tasksRepo`, `projectsRepo`, `objectivesRepo` (single commit por regla F29.G6). `notesRepo` necesita cast `as SaveQueue<NoteRow>` desde el singleton tipado con la intersección — seguro porque `content` es opcional. Verificado inline: `inboxRepo.convertToNote/Task/Project` solo usa el id retornado para `markProcessed`, no lee el doc del server (sin race con queue async). +1 test específico de `createFromInbox` verificando que `content` extra-schema viaja por el queue al `setDoc(merge:true)`. Archivos tocados: los 4 repos + `src/infra/repos/notesRepo.test.ts`.
- **F30.4 — useNote subscribe + page.tsx dual-fix (G3'):** `useNote` se suscribe al `saveNotesCreatesQueue` via `useSyncExternalStore`; el `createEntryStatus` entra a las deps del useEffect getDoc — cuando un create transitiona pending → synced → undefined (GC), el effect re-corre y trae el doc real automáticamente sin requerir navegación. `page.tsx` aplica el cross-check en DOS lugares: el useEffect redirect (línea 75) Y la condition del skeleton (línea 80). Sin tocar el skeleton, `notFound=true` dejaba `<NoteEditorSkeleton />` para siempre. Anti-pattern explícito documentado: NO leer `content` desde TinyBase para "optimizar" durante create-pending — la row TinyBase no tiene `content` por schema-strict, solo el payload del queue lo tiene. Archivos tocados: `src/hooks/useNote.ts`, `src/app/notes/[noteId]/page.tsx`.
- **F30.5 — Indicator + handler discard (G4):** `ENTITY_LABELS` extendido con 4 entries para los creates queues alineadas posicionalmente con `allQueues` ("nota nueva" vs "edición de nota", D8). `createsQueueBindings` exportado en `saveQueue.ts` con (queue, store, table) por cada createsQueue. `handleDiscardAll` del `<PendingSyncIndicator />` itera bindings y hace `store.delRow(table, id)` por cada entry pending ANTES de `allQueues.forEach((q) => q.clear())` — sin esto, descartar deja la row local huérfana (G4: el persister no propaga deletes para rows que nunca llegaron al server). Archivos tocados: `src/hooks/usePendingSyncCount.ts`, `src/lib/saveQueue.ts`, `src/components/layout/PendingSyncIndicator.tsx`.
- **F30.6 — QA E2E + cierre:** smoke Playwright validó indicator con label custom "1 nota nueva pendiente", popover con la entry, click "Descartar" → queue vacío + row local borrada inmediatamente sin refresh (G4 ✅) + cero errores en consola tras inyección de los 4 nuevos singletons. Suite completa: 97 tests verde por commit, lint clean, build clean.

## Lecciones

- **Patrón sync→async para creates con UUID local preserva el contrato `Promise<string>` sin sacrificar awaitability.** El factory genera `crypto.randomUUID()` sync, aplica `setRow` sync, encola `setDoc` y retorna el id INMEDIATAMENTE. Generalizable a cualquier flow "navigate-on-create" que actualmente bloquee esperando confirmación de write — el navigate puede ocurrir antes del flush si la página destino tolera la ventana via cross-check al queue.
- **`useSyncExternalStore` con primitivo derivado en deps del useEffect evita re-fetches innecesarios.** Subscribir directamente al queue retorna nueva referencia del entry en cada `patchEntry` (5+ transiciones por flush). Derivar `createEntryStatus = entry?.status` y usarlo como dep significa que el effect solo re-corre cuando el status cambia (pending → retrying → synced → undefined), no en cada notify intermedio. Patrón aplicable a cualquier hook reactivo sobre fuentes externas con muchas transiciones.
- **`updateDoc` falla con `not-found` durante create-pending; `setDoc(merge:true)` no.** Si un flow encolará el create pero el editor permite escritura inmediata, cualquier write subsiguiente debe usar `setDoc(merge)` en lugar de `updateDoc`. Aplica a cualquier feature futura con "create + escritura inmediata" pattern.
- **TinyBase v8 `onSnapshot` post-reconnect puede borrar rows huérfanas que nunca alcanzaron el server.** El persister re-llama `getPersisted` → `getDocs(col)` retorna snapshot sin el doc nuevo → diff aplica `delRow` local. Mitigación: el cross-check de la página destino al queue distingue "doc real borrado" de "create error pending retry". Aplica a cualquier flow async-write cuando el SDK de Firestore no retiene el pending write tras error sincrono (rules-deny, etc.).
- **Anti-pattern: leer payloads del queue para "optimizar" UI.** Tentación recurrente: si TinyBase no tiene `content` durante create-pending, ¿por qué no leerlo del `getEntry().payload`? Costo: complejidad, edge cases por status, acoplamiento entre layer de UI y layer de queue. Mejor aceptar UX subóptimo (editor vacío hasta flush + subscribe rehidrata) y mantener el queue como caja negra para los consumers.
- **`createsQueueBindings` exportado en `saveQueue.ts` es el patrón canónico para handlers que necesitan orquestar `delRow` además de `clear()`.** Centraliza el mapeo (queue → store, table) en un solo lugar. Cualquier handler futuro que descarte creates (no solo el indicator) debe iterar este array. Si en el futuro un componente del UI consume `cancel(id)` directo sobre un createsQueue, debe replicar el `delRow` manualmente — el cancel del queue NO toca el store (separación de capas intencional).
- **`createFirestoreRepo` con dos queues opcionales independientes (`queue?` para meta + `createsQueue?` para creates) escala mejor que un único queue compartido.** Tipos distintos (`Partial<Row>` vs `Row`), upserts no chocan, retro-compat preservada. Cualquier nueva categoría de write (ej. bulk imports en F31+) puede sumar un tercer queue opcional sin tocar las existentes.
